### 1.2 核心差距（本次规划重点）

**差距 A — 生产链路未走 DAG，非风场图层实际是模拟数据**

调度顺序见 [workflow_tasks.py](file:///d:/temp_desktop/Proj/Comprehensive%20Geographic%20Data%20Analysis%20system/Code/backend/app/tasks/workflow_tasks.py#L50-L59) 的 `_BRIDGE_CHAIN`：
```
gee → weather_bridge_service → algorithm → weather_engine_service(fallback)
```
- `weather_bridge_service.supports()` 要求 `payload.weather_request.workflow` 存在才接管。
- 前端 `runWorkflowForCatalog`（[index.ts L894-917](file:///d:/temp_desktop/Proj/Comprehensive%20Geographic%20Data%20Analysis%20system/Code/frontend/src/stores/layers/index.ts#L894-L917)）提交时**不带 `weather_request`**。
- 结果：所有天气图层都落到 `weather_engine_service.execute()` 的 layer-based fallback。
- 在该 fallback 中，只有 **wind-field** 走 `fetch_grid_forecast`（真实网格），**temperature/precipitation/humidity/pressure/visibility 全部用 `_*_value_for_location` 数学公式模拟**（见 [service.py L456-551](file:///d:/temp_desktop/Proj/Comprehensive%20Geographic%20Data%20Analysis%20system/Code/backend/app/weatherengine/service.py#L456-L551) 与 L1511-1596）。

即：`GridFetchNode → *GridRenderNode` 这条真实数据 DAG 已完整实现，但**从未被生产流量触发**。旧调试计划 P3（"非风场用模拟数据"）的根因就在这里，且当前仍然存在。

**差距 B — service.py fallback 中非风场图层未复用 `fetch_grid_forecast`**

即便不启用 DAG，fallback 分支里 temperature/precipitation 等也可以像 wind-field 一样调用 `fetch_grid_forecast` + `build_*_geojson_from_grid`，但目前没有。

**差距 C — 节点端口声明不一致（小 bug）**

[wind_field_render.py](file:///d:/temp_desktop/Proj/Comprehensive%20Geographic%20Data%20Analysis%20system/Code/backend/app/weatherengine/nodes/wind_field_render.py#L118-L131) 的 `build_spec()` 缺少 `grid_data`、`layer_id`、`viewport_bbox`、`bbox` 输入端口声明，而其余 5 个渲染节点都已声明。虽然 `execute()` 用 `inputs.get("grid_data")` 仍能取到值，但 spec 不一致会导致前端编辑器 / DAG 连线校验缺端口。
