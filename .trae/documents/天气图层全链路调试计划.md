## 概述

对天气图层从数据获取到前端显示的完整链路进行系统性调试，添加丰富的调试输出，验证每个环节的数据正确性，修复发现的问题。

## 当前状态分析

### 数据链路

```
前端添加图层 → runWorkflowForCatalog(layer_id, lat/lng, viewport_bbox)
  → 后端 WeatherEngineService.execute()
    → _resolve_point() 解析中心点
    → _resolve_render_bbox() 解析渲染范围
    → 风场: fetch_grid_forecast() 从 Open-Meteo 获取真实网格数据
    → 温度/降水等: 仅获取中心点数据，用数学公式插值生成网格（模拟数据）
    → build_wind_geojson_from_grid() / build_temperature_geojson() 生成 GeoJSON
    → result_storage_service 存储 artifact
  → 前端 pollWorkflowRun() 轮询状态
    → buildJobLayer() 提取 mapLayerPayload（renderHint, geojsonUrl）
    → MapCanvas watch 触发 syncWeatherOverlay()
      → particle_flow: syncWindParticleFlow() → WindParticleCanvas (Canvas 2D)
      → grid_fill: syncWeatherGridFillOverlay() → MapLibre GeoJSON fill
      → point_symbol: syncWeatherPointOverlay() → MapLibre circle
```

### 已识别的疑似问题

| # | 问题 | 位置 | 严重度 |
|---|------|------|--------|
| P1 | 风场只显示一小片区域 — viewport_bbox 可能未正确传递或默认范围太小（±1.6°/±1.2°） | 后端 `_resolve_render_bbox` L1442-1470 | 高 |
| P2 | 粒子流只是好几条线条闪动 — 粒子数/速度比例/网格分辨率可能不合理 | 前端 `wind-particle-canvas.ts` L60-66 | 高 |
| P3 | 非风场图层（温度/降水/气压/湿度/能见度）使用数学公式插值而非真实 API 数据 | 后端 `service.py` build_temperature_geojson 等 L694+ | 中 |
| P4 | 频繁缩放可能导致重复提交工作流 — 防抖逻辑可能不完善 | 前端 `index.ts` refreshActiveWeatherWorkflows L596-610 | 中 |
| P5 | Open-Meteo 多点请求格式 — 批量请求时 lat/lon 数组的构建逻辑可能不正确 | 后端 `client.py` L217-228 | 中 |
| P6 | 粒子流 canvas 偏移计算 — `computeCanvasLayout` 可能未正确处理地图投影 | 前端 `webgl-utils.ts` | 中 |
| P7 | 温度等图层的 COG 预览生成可能失败（无 cog_preview_url） | 后端 `_build_temperature_cog_artifact` | 低 |

## 实施计划

### 阶段 1：添加调试输出（只读探针，不改逻辑）

在每个关键环节添加带时间戳和组件名的调试日志，用于诊断数据流。

#### 1.1 后端调试输出

**文件：`Code/backend/app/weatherengine/service.py`**

在以下位置添加调试日志：
- `execute()` 方法入口：打印 layer_id, latitude, longitude, viewport_bbox, parameters
- `_resolve_render_bbox()`：打印输入的 viewport_bbox 和最终返回的 bbox
- `_build_map_layer_outputs()`：打印生成的 GeoJSON feature 数量、bbox 范围、首个/末个 feature 坐标
- `build_wind_geojson_from_grid()`：打印网格 rows/cols、lats/lons 范围、speed/direction 样本值

**文件：`Code/backend/app/weatherengine/client.py`**

在以下位置添加调试日志：
- `fetch_grid_forecast()` 入口：打印 bbox, resolution, rows, cols, total_points
- HTTP 请求前后：打印请求 URL 长度、响应状态、响应大小
- 解析响应后：打印 current 数据字段名和值数量

#### 1.2 前端调试输出

**文件：`Code/frontend/src/stores/layers/index.ts`**

在以下位置添加调试日志：
- `runWorkflowForCatalog()`：打印 catalogId, parameters, map_context
- `pollWorkflowRun()` 成功时：打印 status, result_refs 数量, mapLayerPayload 概要
- `refreshActiveWeatherWorkflows()`：打印触发的图层列表和跳过原因

**文件：`Code/frontend/src/components/MapCanvas.vue`**

在以下位置添加调试日志：
- `resolveAllWeatherOverlayStates()`：打印每个图层的 renderHint.paint_mode, geojsonUrl, cogPreviewUrl
- `syncWindParticleFlow()`：打印 geojsonUrl, fetch 结果 feature 数量, grid bounds
- `syncWeatherGridFillOverlay()`：打印 geojsonUrl, fill 颜色配置

**文件：`Code/frontend/src/components/map/wind-particle-canvas.ts`**

在以下位置添加调试日志：
- `buildWindGridFromGeoJSON()`：打印 grid rows/cols/bounds
- `initParticles()`：打印粒子数量、grid 范围
- `draw()` 首次：打印粒子移动量、插值风速样本

### 阶段 2：逐环节验证数据正确性

#### 2.1 验证数据获取

通过 API 测试脚本验证：
- 提交 wind-field 工作流，检查返回的 GeoJSON：
  - feature 数量是否合理（viewport 越大点越多）
  - 坐标范围是否匹配 viewport_bbox
  - speed/direction 值是否为真实数据（非全 0 或全相同）
- 提交 temperature 工作流，检查是否使用了模拟数据
- 检查 Open-Meteo API 响应格式是否正确

#### 2.2 验证工作流处理

- 检查前端传递的 viewport_bbox 是否正确（当前地图视口）
- 检查分辨率动态调整逻辑是否合理
- 检查缓存命中/未命中情况
- 验证 hour 参数是否被正确使用

#### 2.3 验证生成数据

- GeoJSON 坐标范围是否匹配请求的 bbox
- feature 数量是否在合理范围（风场 100-200 点）
- 时间戳是否正确（current 时间）
- 网格点是否均匀分布

#### 2.4 验证前端显示

- Vite proxy 是否正常转发 `/artifacts/*` 请求
- GeoJSON fetch 是否成功
- WindParticleCanvas 网格构建是否正确
- MapLibre fill/line 图层是否正确渲染
- canvas 叠加层 z-index 是否正确

### 阶段 3：修复发现的问题

根据阶段 2 的验证结果，修复具体问题。预期修复方向：

1. **P1 — 风场显示区域小**：
   - 检查前端 `currentMapBBox` 是否正确更新
   - 确认 `viewport_bbox` 在工作流提交时被正确传递
   - 可能需要增大默认 bbox 范围

2. **P2 — 粒子流效果差**：
   - 调整 `speedScale` 参数（当前 0.00012 可能太小）
   - 检查 `computeCanvasLayout` 是否正确计算 canvas 偏移
   - 验证粒子在网格范围内的分布和移动

3. **P3 — 非风场图层使用模拟数据**：
   - 将温度/降水等图层也改为使用 `fetch_grid_forecast` 获取真实网格数据
   - 或者明确标注为"估算数据"

4. **P4 — 频繁缩放问题**：
   - 检查防抖逻辑（VIEWPORT_DEBOUNCE_MS = 500）
   - 确认 `refreshActiveWeatherWorkflows` 跳过已成功的工作流
   - 考虑添加缩放级别变化阈值

5. **P5 — Open-Meteo 多点请求**：
   - 检查批量请求的 lat/lon 数组构建逻辑
   - 验证响应解析是否正确处理多点格式

### 阶段 4：回归测试

- 修改后重新验证所有天气图层
- 确保风场粒子流、温度 grid_fill、降水等图层都能正常显示
- 验证缩放/平移后图层数据是否正确更新
- 确保不会引入新的问题

## 假设与决策

1. **调试输出格式**：统一使用 `[组件名]` 前缀 + 时间戳，便于过滤
2. **调试输出级别**：后端使用 `logger.info`，前端使用 `console.log`（带前缀）
3. **不修改现有逻辑**：阶段 1 只添加调试输出，不改变任何处理逻辑
4. **渐进式修复**：每修复一个问题后验证，避免引入新问题
5. **非风场图层的模拟数据**：暂不改为真实数据（工作量较大），先确保风场正确
