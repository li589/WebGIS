# 架构优化双层方案（2026-07-12）

## Summary

本方案面向当前仓库的真实结构，目标不是“重写系统”，而是在保留现有 `workflow-runs`、天气链路、GEE、Python provider 主线能力的前提下，把系统逐步收敛到更结构化、易修改、可扩展的架构。

本次规划采用“双层方案”：

1. **目标架构蓝图**：明确未来统一方向  
   - 一切地图展示数据最终都可通过标准化的 `XYZ/Tile Service` 能力消费
   - 工作流负责生产数据、更新数据、导出结果，而不是直接承担所有展示协议
   - 图层能力以 `capabilities` 驱动，而不是以 `layer_id.startsWith(...)` 驱动
   - 引擎接入通过注册表机制管理，而不是继续堆积 `if/elif`
   - 契约以后端为单一事实源，前端类型自动生成

2. **首阶段可执行落地路线**：优先做 `Capabilities + 渲染注册表`
   - 先降低“新增一个图层/天气层要改很多处”的成本
   - 再拆前端大组件/大 store
   - 再收口后端 engine request populators / readiness / bridge
   - 最后再统一瓦片服务和契约生成链路

## Current State Analysis

### 1. 当前系统的优点

- 已有明确的前后端边界：  
  - 前端主入口：`Code/frontend/src/views/DashboardView.vue`
  - 后端主入口：`Code/backend/app/main.py`
  - 共享契约：`Code/shared/contracts/api_contracts.py`
- 已形成统一工作流主线：`/workflow-runs -> interaction_hub -> tasks/services`
- 已存在控制流 / 数据流分离思路：  
  - 状态/事件：`/workflow-runs/{run_id}`、`/events`
  - 展示结果：`/view`、`/artifacts/*`
- 天气链路、GEE、Python provider 都已进入真实工程状态，而非空骨架

### 2. 当前最主要的结构问题

#### 2.1 前端核心模块职责过重

- `Code/frontend/src/components/MapCanvas.vue`  
  当前约 `2013` 行，混合承担：
  - 地图实例生命周期
  - source/layer 管理
  - 天气渲染
  - 视口回写
  - 交互事件派发

- `Code/frontend/src/stores/layers/index.ts`  
  当前约 `1404` 行，混合承担：
  - 图层目录加载
  - workflow 提交/轮询/取消/重试
  - 视口状态
  - 点天气查询
  - 粒子流控制
  - 与 `weather-tile-manager` 协调

这会导致地图行为、工作流行为、图层元数据、渲染策略强耦合。

#### 2.2 图层能力依赖硬编码判断

- `Code/frontend/src/components/map/weather-render.ts`
- `Code/frontend/src/stores/layers/index.ts`
- `Code/frontend/src/services/weather-tile-api.ts`
- 以及若干 MapCanvas 内部逻辑

当前大量行为仍然依赖：

- `layerId.startsWith('wind-field')`
- `layerId.startsWith('temperature')`
- `catalogId === 'precipitation'`
- `supportsParticleFlow(catalogId)`

这意味着新增图层、改图层渲染模式、改结果交付方式时，必须跨多个文件同步修改。

#### 2.3 后端引擎接入仍然依赖集中分支

- `Code/backend/app/services/workflow_request_resolver.py`

当前按 `descriptor.engine` 做集中分支：

- `python_provider`
- `gee`
- `weather_workflow`

随着引擎类型和执行模式继续增长，这个文件会变成持续膨胀的总开关。

#### 2.4 后端 orchestration 层过大

- `Code/backend/app/services/interaction_hub.py` 当前约 `1454` 行
- `Code/backend/app/api/routes.py` 当前约 `433` 行

`interaction_hub.py` 同时承担：

- submit / cancel / retry
- 状态流转
- failure/retry
- runtime status
- runtime config
- follow-up dispatch

这会让修改某个领域逻辑时，必须在大文件中穿梭，且容易引入侧向回归。

#### 2.5 契约仍然是“双份手写镜像”

- 后端：`Code/shared/contracts/api_contracts.py`
- 前端：`Code/frontend/src/services/runtime-api.ts`

当前前端 TypeScript 类型基本是后端 Pydantic 模型的镜像手写版。短期可用，长期容易漂移。

#### 2.6 展示交付协议还未统一到“瓦片优先 + 标准结果接口”

当前同时存在几种展示交付方式：

- `map_layer.inline_data`
- `geojson_url`
- `cog_preview_url`
- `/artifacts/{artifact_id}/preview.png`
- 新增中的天气瓦片 workflow 链路
- `/weather/tiles/*` 调试直连 REST

这说明系统已经在向“瓦片化展示”演进，但还未把它收口成正式统一架构。

## Target Architecture

### 1. 核心原则

1. **工作流负责生产，瓦片服务负责展示**
2. **图层消费前先看 capabilities，而不是猜 layer_id**
3. **引擎接入通过注册表管理**
4. **渲染器按 strategy / paint_mode 注册**
5. **契约以后端 schema 为单一事实源**

### 2. 目标分层

#### 2.1 数据生产层

由工作流负责：

- GEE workflow
- weather workflow
- python provider workflow
- 后续 NDVI / SMAP / 课题组算法 workflow

统一输出以下一种或多种结果能力：

- 可供瓦片服务消费的数据源
- 标准 artifact（csv、mat、png、parquet、json 等）
- 标准 view/result 接口摘要

#### 2.2 数据交付层

统一抽象为三类 delivery mode：

- `xyz_raster_tiles`
- `xyz_vector_tiles`
- `artifact_download`

可保留一个过渡态：

- `inline_map_layer`

但只作为兼容，不作为长期主展示协议。

#### 2.3 能力描述层

由 `/layers` 返回每个图层的 `capabilities`，至少包含：

- `data_domain`
- `engine_type`
- `render_strategy`
- `delivery_modes`
- `supports_particle_flow`
- `paint_modes`
- `workflow_bootstrap_required`
- `tile_endpoint_template`（如果存在）
- `result_interfaces`

#### 2.4 前端渲染层

前端不再直接按 `layer_id` 决定渲染，而是：

- 先读 `capabilities.render_strategy`
- 再选对应 renderer
- `paint_mode` 由渲染器注册表解释

#### 2.5 引擎接入层

后端建立统一注册表：

- `EngineRequestPopulator`
- `EngineReadinessResolver`
- `EngineExecutionBridge`

每个 engine 独立注册，不再在 resolver 中做集中 if/elif 扩展。

## Proposed Changes

### Phase 1：Capabilities + 渲染注册表（首阶段，优先）

> 目标：先让“新增图层/调整渲染方式”从多点硬编码，变成“补一条元数据 + 注册一个 renderer”。

#### 1. 后端扩展 `/layers` 契约，返回正式 `capabilities`

**主要文件**

- `Code/shared/contracts/api_contracts.py`
- `Code/backend/app/services/layer_catalog.py`
- `Code/backend/app/api/routes.py`
- `Code/backend/app/weatherengine/constants.py`

**改动内容**

1. 在 `LayerDescriptor` 中新增或收口能力字段
   - `engine_type`
   - `data_domain`
   - `render_strategy`
   - `delivery_modes`
   - `paint_modes`
   - `supports_particle_flow`
   - `workflow_bootstrap_required`
   - `tile_endpoint_template`
   - `result_interfaces`

2. 把天气层、provider 层、后续 GEE 层的展示能力统一放到 `layer_catalog.py` 生成，不让前端再自行推断

3. `weatherengine/constants.py` 中与 layer 类型相关的知识，整理成可被 catalog 消费的元数据，而不只服务天气构建逻辑

**为什么这样改**

- 现在真实问题不是“没有元数据”，而是元数据不够表达展示能力
- 只要 `/layers` 能表达图层能力，前端渲染逻辑就能从硬编码判断迁移出去

#### 2. 前端建立渲染注册表，而不是散落在 store / component 里判断

**主要文件**

- `Code/frontend/src/components/map/weather-render.ts`
- `Code/frontend/src/stores/layers/index.ts`
- `Code/frontend/src/components/MapCanvas.vue`

**新增目录建议**

- `Code/frontend/src/map/render-registry/`
  - `types.ts`
  - `render-strategies.ts`
  - `paint-mode-registry.ts`
  - `layer-capability-utils.ts`

**改动内容**

1. 新增 `render_strategy -> renderer` 注册表
   - 例如：
     - `inline_geojson`
     - `xyz_vector_tiles`
     - `xyz_raster_tiles`
     - `artifact_preview_raster`
     - `particle_flow_overlay`

2. 新增 `paint_mode -> style builder` 注册表
   - 当前 `weather-render.ts` 中的颜色、图例、opacity、point radius 等逻辑，迁移为按 `paint_mode` 或 `strategy` 注册

3. `layers/index.ts` 不再直接以 `catalogId` 判断支持粒子流、地图型结果、天气型结果；改为读 `capabilities`

4. `MapCanvas.vue` 只负责调度 renderer，不再直接理解所有 layer family 的细节

**为什么这样改**

- 这是最快降低前端修改成本的地方
- 也是后续统一天气层、底图、GEE 结果图层、NDVI/SMAP 图层的必要前提

#### 3. 保留兼容层，不一次性推翻现有数据面

**改动位置**

- `Code/frontend/src/stores/layers/result-adapter.ts`
- `Code/backend/app/services/result_view_service.py`

**改动内容**

1. `inline_map_layer` 作为兼容策略保留
2. 新 registry 层先兼容现有 `map_layer.payload.render_hint` 和 `layer_assets`
3. `result_view_service.py` 只保持摘要职责，不承担展示策略分发

**为什么这样改**

- 当前已有天气链路、provider 链路在跑，不能一次性替换结果面协议

### Phase 4：统一 XYZ Tile Service 架构

> 目标：把“展示路径”和“生产路径”彻底分清楚。

#### 1. 明确 Tile Service 是正式展示协议

**主要文件**

- `Code/backend/app/api/tile_routes.py`
- `Code/backend/app/api/weather_tile_routes.py`
- `Code/backend/app/services/tile_proxy_service.py`
- `Code/backend/app/weatherengine/tile_service.py`

**改动内容**

1. 抽象统一 `TileServiceDescriptor`
2. 区分：
   - 底图瓦片
   - 天气生成瓦片
   - workflow 产物瓦片

3. 后续 NDVI / SMAP / 课题组结果图层走同一类 tile contract，而不是每类单独定义地图资产协议

#### 2. Workflow 与 Tile 的关系标准化

**标准关系**

- workflow：负责生成结果、刷新缓存、落盘、产出 manifest
- tile service：负责按 `x/y/z` 或等价 tile key 暴露数据
- 结果接口：负责表格、mat、csv、png、诊断信息等非瓦片结果

**新增接口方向**

- `result_interfaces` 中显式标识：
  - `tiles`
  - `artifact`
  - `table`
  - `image`
  - `binary`
  - `diagnostic`

#### 3. 天气瓦片链路正式收口

**主要文件**

- `Code/frontend/src/stores/weather-tile-manager.ts`
- `Code/frontend/src/services/weather-tile-api.ts`
- `Code/backend/app/api/weather_tile_routes.py`
- `Code/backend/app/weatherengine/tile_service.py`

**改动内容**

1. 明确当前主链：  
   - 生产通过 workflow
   - 展示通过 tile service / tile cache

2. `/weather/tiles/*` 从“调试直连”过渡为正式能力，或者明确废弃，仅保留 workflow-backed tile 模式

3. 前端 tile manager 不再自己知道天气特殊性，而是作为通用 tile orchestration 组件演进

## Assumptions & Decisions

### 已确认决策

1. 本次规划采用“双层方案”
2. 首阶段优先：`Capabilities + 渲染注册表`
3. 契约自动生成路线：`OpenAPI -> TS`
4. 长期方向：展示能力优先收口到 `XYZ/Tile Service`
5. 工作流是生产/刷新/导出机制，而不是唯一展示协议
6. 长期保留类似 ComfyUI 的模块编排愿景：  
   - GEE 模块集
   - 开放天气模块集
   - Python import 模块集

### 规划中的关键假设

1. 现有 `workflow-runs` 主线必须保留，不做推翻式替换
2. 当前天气链路、provider 链路、GEE 链路都要允许渐进迁移
3. 第一阶段允许兼容 `inline_map_layer` 与现有 `render_hint`
4. 瓦片服务统一化是中期主目标，不要求第一阶段全部完成

### 明确不在首阶段解决的内容

1. 不立即实现完整 ComfyUI 前端编排器
2. 不立即把所有底图/叠加图层全部改成统一瓦片服务
3. 不立即移除现有天气/工作流兼容路径
4. 不在第一阶段强制替换全部手写 TS 类型

## 最终结论

当前项目最应该避免的是继续把新能力直接叠加到：

- `MapCanvas.vue`
- `stores/layers/index.ts`
- `workflow_request_resolver.py`
- `interaction_hub.py`

真正正确的方向是：

1. **先让 `/layers` 能表达图层能力**
2. **再让前端按 capability 走注册表**
3. **再让后端按 engine 注册表演进**
4. **最终把展示能力统一到 tile service，把生产能力统一到 workflow，把非地图结果统一到标准结果接口**

这条路线和你提出的目标是一致的：  
系统会逐步朝着“模块集 + 工作流生产 + 标准交付接口 + 前端按能力消费”的方向收口，而不是继续依赖散落的条件判断和局部特例。
