## 规划原则

1. **先提交后改进** — 将已验证的 MapCanvas 重构成果作为 baseline 提交，避免与架构改进混合
2. **依赖顺序** — 按依赖关系分阶段执行，前置阶段为后续阶段提供稳定基础
3. **每阶段可验证** — 每阶段完成后运行全量测试（前端 vitest + vue-tsc + vite build；后端 pytest）
4. **向后兼容** — 重构期间保持 API 契约不变，内部重组通过 re-export 保持外部调用点稳定

_ENGINE_POPULATORS: list[EngineRequestPopulator] = []

def register_engine_populator(populator: EngineRequestPopulator) -> None: ...
def get_engine_populator(engine_name: str) -> EngineRequestPopulator | None: ...
def iter_engine_populators() -> Iterator[EngineRequestPopulator]: ...

## 执行顺序与依赖

```
Phase 0 (提交 baseline)
  ↓
Phase 1 (M1 前端清理) ──────────────────────────┐
  ↓                                              ↓
Phase 2 (Engine 注册体系) ──→ Phase 3 (hub 拆分) → Phase 4 (routes 拆分)
                                                 ↓
                                          Phase 5 (M4 统一 tile)
                                                 ↓
                                          Phase 6 (M5 契约生成)
```

- Phase 1 与 Phase 2/3/4 可并行（前端 vs 后端）
- Phase 3 必须在 Phase 4 之前（routes 调用 hub 方法）
- Phase 5 依赖 Phase 4（统一路由需要 router 拆分完成）
- Phase 6 依赖 Phase 4（需要稳定的 OpenAPI 端点集合）

## 假设与决策

1. **向后兼容优先**: 所有重构通过 facade/re-export 保持外部接口不变，避免破坏性变更
2. **测试先行**: 每阶段完成后必须通过全量测试才进入下一阶段
3. **`primary_metric_field` 新字段**: 这是 Phase 1 的关键决策 — 将 InfoPanel.vue 的 catalogId→字段名映射正式化为后端元数据，而非简单删除
4. **interaction_hub facade 保留**: 不删除 `InMemoryInteractionHub` 类名，避免 routes.py 大量改动
5. **M4 兼容路由保留**: 旧路由暂不删除，前端迁移完成后再清理
6. **M5 渐进迁移**: 生成的类型先与手写类型共存，逐步替换
