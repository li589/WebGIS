# WebGIS 后端开发者协作说明

这份文档给负责后端开发的同学使用。

当前项目约定：

- 仓库地址：`https://github.com/li589/WebGIS`
- 稳定分支：`main`
- 日常开发分支：`dev`
- 你主要负责目录：`Code/backend`、`Code/algorithms`
- 对方主要负责目录：`Code/frontend`
- 双方共同维护目录：`Code/docs`、`Code/shared`、`README.md`

说明：早期约定中的 `Code/infra` 目录当前不存在；基础设施以 `Code/backend/docker-compose.yml` 与根目录 `launch.py` 为准。

## 第一次获取项目

第一次在本地使用这个项目时：

```bash
git clone https://github.com/li589/WebGIS.git
cd WebGIS
git checkout dev
git pull origin dev
```

## 你平时主要改哪里

你主要修改：

- `Code/backend`
- `Code/algorithms`
- 仓库根启动脚本 / `Code/backend/docker-compose.yml`（基础设施）

尽量不要主动改这些目录，除非已经和前端同学确认：

- `Code/frontend`

共同文件如果要改，最好先沟通：

- `Code/docs`
- `Code/shared`
- `README.md`

## 每次开始开发前怎么同步

由于你主要做后端，不希望覆盖自己正在开发的后端代码，所以推荐先按目录同步前端和公共文件，而不是直接整分支 `pull`。

建议执行：

```bash
git checkout dev
git fetch origin
git restore --source origin/dev -- Code/frontend
git restore --source origin/dev -- Code/shared Code/docs README.md Code/README.md
```

这样做的效果是：

- 更新前端代码
- 更新共享协议和文档
- 尽量不动你自己的后端目录

## 如果你当前后端还没提交

在同步前先检查状态：

```bash
git status
```

如果你本地还有重要后端改动，建议先做其中一种：

### 方案 1：直接提交

```bash
git add .
git commit -m "wip(backend): current progress"
```

### 方案 2：临时暂存

```bash
git stash
```

恢复时：

```bash
git stash pop
```

## 你日常开发和提交的推荐流程

```bash
git checkout dev
git fetch origin
git restore --source origin/dev -- Code/frontend
git restore --source origin/dev -- Code/shared Code/docs README.md Code/README.md
```

然后在 `Code/backend`、`Code/algorithms` 中开发。

开发完成后：

```bash
git add .
git commit -m "feat(backend): 简要说明本次改动"
git push origin dev
```

提交信息示例：

```bash
git commit -m "feat(backend): add task submit api"
git commit -m "fix(backend): normalize response schema"
git commit -m "feat(algorithms): add demo model loader"
```

## 更推荐的方式

如果你有一块独立后端功能要开发，推荐从 `dev` 拉自己的功能分支：

```bash
git checkout dev
git pull origin dev
git checkout -b feature/backend-task-api
```

开发完成后：

```bash
git add .
git commit -m "feat(backend): complete task api"
git push -u origin feature/backend-task-api
```

然后再由项目负责人合并到 `dev`。

## 什么时候用正常 `git pull`

如果后续你开始同时频繁改这些公共目录：

- `Code/shared`
- `Code/docs`
- 根目录说明文档
- 前后端共同依赖的接口定义

那就不建议只按目录同步了，应改成：

```bash
git checkout dev
git pull origin dev
```

## 后端开发时特别要注意

- 提交前先 `git status`
- 尽量只提交与你当前后端功能相关的文件
- 不要把本地环境、缓存、临时文件提交到仓库
- 如果接口返回结构变化，先同步给前端同学
- 如果新增公共协议，优先更新 `Code/shared` 或 `Code/docs`

## 常用命令速查

切到 `dev`：

```bash
git checkout dev
```

同步前端代码：

```bash
git fetch origin
git restore --source origin/dev -- Code/frontend
```

同步公共文档和共享协议：

```bash
git fetch origin
git restore --source origin/dev -- Code/shared Code/docs README.md Code/README.md
```

查看当前状态：

```bash
git status
```

提交并推送：

```bash
git add .
git commit -m "feat(backend): 本次改动说明"
git push origin dev
```

## 一句话原则

你负责后端，就重点维护 `Code/backend`、`Code/algorithms` 以及本地基础设施配置；同步时优先拉前端和共享目录，不要随手覆盖自己正在做的后端代码。
