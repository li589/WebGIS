# 双通道接口设计总结

## 目标

将后端 API 划分为**控制流（Control Plane）**与**数据流（Data Plane）**两条主线：

- 控制流负责任务编排、状态查询、事件流、取消与重试
- 数据流负责结果视图、artifact 访问、按需加载与展示适配

这样可以减少前端对后端内部 DTO 的直接依赖，降低接口拥堵与契约漂移风险。

## 当前接口分层

### 控制流接口

这些接口维持为稳定的编排与状态面：

- `POST /workflow-runs`
- `GET /workflow-runs/{run_id}`
- `GET /workflow-runs/{run_id}/events`
- `POST /workflow-runs/{run_id}/cancel`
- `POST /workflow-runs/{run_id}/retry`
- `GET /runtime/status`

控制流当前还承载两类稳定语义：

- `ExecutionStatus.retry_pending`：表示任务因瞬态错误进入等待自动重试的中间态
- `FailureCategory`：表示失败的分类语义，供 bridge、workflow 服务和前端状态展示统一理解

特征：

- 响应轻量
- 频繁轮询友好
- 适合状态机与调度逻辑
- 不承载大结果或复杂展示逻辑

### 数据流接口

这些接口负责结果展示和资源访问：

- `GET /workflow-runs/{run_id}/view`
- `GET /artifacts/{artifact_id}`
- `GET /artifacts/{artifact_id}/preview.png`
- `GET /unified-tiles/{layer_id}/{z}/{x}/{y}`（底图栅格瓦片入口）
- `GET /weather/tiles/{layer_id}/{z}/{x}/{y}`（天气 GeoJSON 瓦片入口）
- `GET /runtime/tiles/providers`、`/runtime/tiles/cache/*`（底图代理管理面，非数据面）

特征：

- 以展示为中心
- 返回稳定的 UI 视图模型，或按需加载的瓦片 / artifact
- 只暴露必要摘要、指标、结果链接和资源引用
- 适合前端结果展示、懒加载与地图瓦片渲染
- 允许大体量 raster 结果通过 preview route 转成前端可直接消费的 PNG 预览
- 天气层消费 `/weather/tiles`；底图栅格消费 `/unified-tiles`；旧 `/tiles/{provider}/...` 已移除
- 系统设置走控制面 `/config/*`（API Key / 天气源 / GEE），与瓦片数据面分离；密钥明文不经 `/runtime/api-config` 下发
- 工作流事件为 `GET /workflow-runs/{id}/events` JSON 轮询（非 SSE）

## 统一结果视图模型

后端新增了专用于前端展示的 view model：

- `WorkflowRunViewSummaryRow`
- `WorkflowRunViewResponse`

该模型的核心字段包括：

- `run_id`
- `category`
- `title`
- `subtitle`
- `status_text`
- `progress_text`
- `summary`
- `metric_rows`
- `result_url`
- `artifact_refs`
- `can_show_link`
- `updated_at`

设计原则：

- 结果 DTO 保持后端内部语义
- 前端只消费 view model
- view model 以展示稳定性优先，不暴露内部实现细节

## 前端消费边界

前端消费路径已调整为：

- `runtime-api.ts`：负责 workflow / runtime 控制流请求
- `weather-tile-api.ts` + `weather-tile-manager.ts`：负责 `/weather/tiles` 天气 GeoJSON 瓦片请求与调度
- `unified-tile` 相关模块：负责 `/unified-tiles` 底图栅格瓦片请求
- `stores/layers/result-adapter.ts`：把 `WorkflowRunStatusResponse` 与 `WorkflowRunViewResponse` 适配为 job layer
- `components/info-panel/result-adapter.ts`：把 view model 适配为 InfoPanel 展示模型
- `InfoPanel.vue`：只渲染展示模型，不解析后端内部 DTO
- `DashboardView.vue`：只承担布局和状态编排
- `types/api-contracts.ts`：由 OpenAPI 生成，与后端契约对齐

对于天气图层这类“展示即结果”的场景，前端当前还会消费状态接口中的稳定结果引用：

- `map_layer.payload.render_hint`
- `map_layer.payload.layer_assets`
- `map_layer.payload.point_feature`

其中：

- `render_hint` 负责 palette、opacity、paint_mode、legend_ticks 等展示契约
- `layer_assets` 负责 `geojson_url / cog_url / cog_preview_url / cog_bbox`
- `point_feature` 负责点要素或风场采样点的即时渲染输入

## 结果传输策略

### 状态接口

状态接口只提供：

- 任务当前状态
- 进度
- 消息
- 结果引用
- 诊断信息

避免在状态接口中嵌入大体量结果数据。

但当前存在一个刻意保留的例外：`ResultKind.map_layer` 允许携带轻量级展示描述，因为地图图层需要尽快进入前端渲染链；真正的大对象仍通过 artifact 引用按需访问。

### 结果接口

结果接口提供：

- 标题、摘要、状态文本
- 指标行
- 结果链接
- artifact 引用

如果结果较大，应先落盘为 artifact，再通过引用访问。

对于栅格图层，当前数据流已经形成多层读取策略：

1. 先通过 `map_layer.layer_assets` 获取资源描述
2. 天气可视范围按 z/x/y 走 `/weather/tiles`（GeoJSON）
3. 底图栅格按 z/x/y 走 `/unified-tiles`
4. 如前端不能直接消费 COG，则访问 `/artifacts/{artifact_id}/preview.png`
5. 地图侧以 image source + raster layer 或 Canvas overlay 方式叠加展示

## 设计收益

1. **减少耦合**
   - 前端不再直接依赖后端内部结果 DTO 字段

2. **减少拥堵**
   - 控制流接口保持轻量，避免被结果负载拖慢

3. **更利于演进**
   - 后端可以独立演进结果结构，前端通过 view model 保持稳定

4. **更利于测试**
   - 控制流与数据流可以分别做契约测试和行为测试

5. **更利于异构结果接入**
   - 同一条控制流可以同时承接文本、表格、GeoJSON、COG 和前端渲染提示

## 推荐后续方向

- 继续把前端对结果的消费收敛到 view model 与 OpenAPI 类型
- 为 `WorkflowRunViewResponse` 继续补充契约测试和样例文档
- 为 `map_layer` 与 `/weather/tiles`、`/unified-tiles` 数据面补独立契约说明，固定 `render_hint` / `layer_assets` / 瓦片响应字段
- 把 `retry_pending` / `FailureCategory` 的状态机说明补进运行态文档
- 如结果类型继续增长，可进一步按业务域拆分 view model，但保持 view 层统一入口

## 结论

双通道设计的核心不是“多做几个接口”，而是把**状态编排**和**结果展示**拆开：

- 控制流稳定、轻量、可频繁调用
- 数据流稳定、聚合、可展示、可按需加载

这可以显著降低前后端耦合，并让系统更适合持续扩展。
