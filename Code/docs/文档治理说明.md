# 文档治理说明

## 1. 目的

本文档用于统一本仓库的文档维护原则、文档分层方式、更新顺序和归档规则，确保后续在新增功能、调整架构或推进专题工作时，文档能够与代码保持一致。

本说明适用于：

- 项目总览文档
- 工程架构文档
- 协议与接口文档
- 工作流与执行文档
- 专题设计文档
- 阶段记录与验证文档
- 协作文档

## 2. 文档治理目标

文档治理的目标不是“把文档写多”，而是：

1. 让新成员能快速理解项目。
2. 让开发成员能找到当前有效事实源。
3. 让专题文档不污染主架构叙事。
4. 让历史记录可追溯但不混淆当前实现。
5. 让每次架构演进都能同步到文档体系。

## 3. 文档分层

当前仓库文档按六层治理：

```text
入口层 -> 架构层 -> 协议层 -> 执行层 -> 专题层 -> 附录层
```

### 3.1 入口层

用于回答“项目是什么、工程怎么分、先看哪里”。

包括：

- `README.md`
- `Code/README.md`

### 3.2 架构层

用于回答“前端、后端、算法包、共享协议如何协作”。

包括：

- `Code/frontend/README.md`
- `Code/backend/README.md`
- `Code/shared/contracts/README.md`
- `Code/algorithms/providers/Python/README.md`
- `Code/algorithms/providers/docs/detailed_design.md`

### 3.3 协议层

用于回答“前后端、后端与算法包如何对接”。

包括：

- `Code/algorithms/providers/docs/backend_integration_contract.md`
- `Code/algorithms/providers/docs/job_request_http_json_adapter.md`
- `Code/algorithms/providers/docs/workflow_http_json_adapter.md`
- `Code/algorithms/providers/docs/field_mapping_contract.md`
- `Code/algorithms/providers/docs/unified_data_access_design.md`

### 3.4 执行层

用于回答“任务如何进入、如何运行、如何回传”。

包括：

- `Code/algorithms/providers/docs/workflow_extension_design.md`
- `Code/algorithms/providers/docs/pipeline_registry_audit.md`
- `Code/algorithms/providers/docs/platform_queue_worker_integration.md`

### 3.5 专题层

用于承载单个专题、单条链路或单次阶段的设计、验证与结论。

以 `omega_*` 文档为主，另外包括：

- `Code/algorithms/providers/docs/unified_data_access_task_plan.md`

### 3.6 附录层

用于保留历史记录、测试样例、结果样例和阶段结论。

典型内容：

- 验证结果
- 离线运行结果
- 早期设计归档
- 阶段性比较记录

## 4. 文档角色定义

### 4.1 主入口文档

主入口文档是仓库第一阅读入口，负责提供总览和导航。

要求：

- 只讲当前有效事实
- 不堆叠过多历史细节
- 不承担专题实验记录

### 4.2 架构文档

架构文档负责说明系统的分层、职责边界与主数据流。

要求：

- 与当前代码结构一致
- 与模块命名一致
- 与工作流、接口、协议保持一致

### 4.3 协议文档

协议文档负责定义请求、响应、字段、结构和适配方式。

要求：

- 作为接口单一事实源
- 优先使用明确字段表与示例
- 不能与代码实现冲突

### 4.4 执行文档

执行文档负责说明任务如何进入、如何调度、如何运行、如何回传。

要求：

- 明确执行入口
- 明确兼容入口和主入口的区别
- 明确同步、异步、队列和 worker 的关系

### 4.5 专题文档

专题文档负责记录某个方向的设计、验证和阶段结论。

要求：

- 允许包含历史试验过程
- 不作为总架构入口
- 不与主事实文档争夺“当前状态”叙事

### 4.6 附录文档

附录文档用于保存阶段性材料。

要求：

- 明确标记“阶段记录”或“验证结果”
- 不应被默认视为当前规范
- 如结论已被新的主文档吸收，应减少重复描述

## 5. 文档更新原则

### 5.1 代码优先，文档同步

任何涉及以下内容的变化，都应同步检查相关文档：

- 模块名
- 路由名
- 工作流名
- 请求字段
- 输出字段
- 目录结构
- 启动命令
- 环境变量

### 5.2 主文档优先更新

当发生架构变化时，优先更新：

1. `README.md`
2. `Code/README.md`
3. `Code/backend/README.md`
4. `Code/frontend/README.md`
5. `Code/shared/contracts/README.md`
6. `Code/algorithms/providers/Python/README.md`
7. `Code/algorithms/providers/docs/detailed_design.md`

### 5.3 协议先行

如果接口或字段发生变化，应先更新协议文档，再更新调用方说明和示例。

### 5.4 专题文档不反向污染主叙事

专题文档可以保留实验过程，但不要把阶段性尝试写回主入口文档。

### 5.5 旧内容应显式归档

如果某份文档已经不再代表当前事实，应明确标记为：

- 已归档
- 阶段记录
- 验证结果
- 已被新版本替代

## 6. 文档命名规则

### 6.1 主入口与架构文档

建议使用简洁、稳定、长期可维护的名字：

- `README.md`
- `detailed_design.md`
- `backend_integration_contract.md`
- `workflow_extension_design.md`
- `unified_data_access_design.md`

### 6.2 专题文档

专题文档建议保留方向名或专题名，例如：

- `omega_*`
- `unified_data_access_*`
- `platform_*`

### 6.3 阶段记录

阶段记录建议带上语义后缀或日期后缀，例如：

- `*_validation_YYYYMMDD_HHMMSS.md`
- `*_result_YYYYMMDD_HHMMSS.md`
- `*_plan.md`
- `*_checklist.md`

## 7. 文档合并与拆分规则

### 7.1 适合合并的情况

以下情况适合合并：

- 同一主题被多份文档重复描述
- 同一接口被多份文档分别定义
- 同一工作流同时存在“设计版”和“接入版”但内容已趋同

### 7.2 适合拆分的情况

以下情况适合拆分：

- 总览文档过长，开始承载专题细节
- 协议文档开始混入大量运行日志
- 专题文档开始重复主架构叙事
- 阶段记录已演变为常驻规范

### 7.3 处理原则

拆分或合并时，应保持：

1. 主入口文档简洁
2. 架构文档稳定
3. 协议文档精确
4. 专题文档独立
5. 阶段记录可追溯

## 8. 文档一致性检查清单

每次重构或新增内容后，建议检查以下项：

1. 文档中的路径是否与真实代码一致
2. 文档中的模块名是否与当前代码一致
3. 文档中的接口名是否与当前后端一致
4. 文档中的工作流名是否与注册表一致
5. 文档中的请求字段是否与协议一致
6. 文档中的输出字段是否与结果 DTO 一致
7. 文档中的启动命令是否仍可执行
8. 文档中的“当前状态”是否已经过时

## 9. 文档维护流程

建议采用以下流程维护文档：

### 9.1 开发前

- 明确本次变更涉及哪些目录、模块、接口和工作流
- 先列出需要同步的文档

### 9.2 开发中

- 如果接口或结构发生变化，优先同步协议文档
- 如果架构发生变化，优先同步架构文档

### 9.3 开发后

- 检查文档是否与代码一致
- 检查是否产生重复叙事
- 将阶段性内容归档到专题层或附录层

## 10. 文档治理建议

### 10.1 新增功能时

新增功能至少应检查：

- 主入口文档是否需要补一段导航
- 架构文档是否需要补模块说明
- 协议文档是否需要补字段定义
- 专题文档是否需要新增条目

### 10.2 修改接口时

接口修改优先更新：

1. 共享协议
2. 后端说明
3. 前端说明
4. 算法包接入说明

### 10.3 改工作流时

工作流修改优先更新：

1. `backend_integration_contract.md`
2. `workflow_extension_design.md`
3. `pipeline_registry_audit.md`
4. `minimal_job_http_service.md`

### 10.4 改数据接入时

数据接入修改优先更新：

1. `unified_data_access_design.md`
2. `field_mapping_contract.md`
3. 相关专题文档

当前仓库文档已经形成一套较清晰的分层体系：

- 入口层负责导航
- 架构层负责协作边界
