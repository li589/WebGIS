# 代码事实同步文档（2026-07-06）

本文档用于同步当前工作树中的代码事实，范围覆盖 `frontend`、`backend`、`shared` 与 `algorithms/providers/Python`。  
说明：本文档以当前未提交代码为准，不以旧设计文档或历史摘要为准。

## 1. 当前总体状态

当前项目已经从“单一路径 demo WebGIS”演进为多执行链并行的工程状态：

- 前端：以 `DashboardView -> layersStore -> MapCanvas / InfoPanel` 为主链；
- 后端：以 `workflow-runs -> interaction_hub -> workflow_tasks` 为统一调度主链；
- 执行侧：同时存在 `weatherengine`、`weather workflow DAG bridge`、`python provider bridge`、`download real fetch`、`gee bridge`；
- 结果侧：统一返回 `WorkflowResultReference`，前端再通过 `result-adapter` 映射为图层与信息面板展示模型。

当前工作树中，天气链路、失败重试、真实抓取、图层目录与风场高级渲染都在同时推进。

## 2. 当前代码目录中的关键事实

### 2.1 前端

前端主入口仍然是：

- `frontend/src/views/DashboardView.vue`
- `frontend/src/stores/layers/index.ts`
- `frontend/src/components/MapCanvas.vue`
- `frontend/src/components/InfoPanel.vue`
- `frontend/src/components/LayerSidebar.vue`

当前前端事实：

- `LayerSidebar.vue` 已不再是简单图层列表，而是包含：
  - 分类分组
  - 搜索
  - 批量添加
  - 批量显隐
  - 批量移除
  - 拖拽排序
  - 多数据源选择器
- `MapCanvas.vue` 已不再只支持基础 `fill/line/circle/symbol`，还新增三套风场 canvas 叠加层：
  - `components/map/wind-particle-canvas.ts`
  - `components/map/wind-barb-layer.ts`
  - `components/map/wind-contour-layer.ts`
- `MapCanvas.vue` 当前天气渲染语义已从“单图层切换”演进为“多可见天气图层叠加”：
  - `grid_fill` 图层可并行叠加
  - `point_symbol` 图层按各自 source/layer 独立渲染
  - `particle_flow` 图层受 store 独占开关约束，同一时刻只允许一个 catalog 启用
  - `grid_fill` 优先走 `cog_preview_url + cog_bbox` 的 raster preview，缺失时回退 GeoJSON
- `InfoPanel.vue` 已承接：
  - workflow 状态
  - 点天气
  - 天气图例
  - 元数据
  - overlay 图层摘要
- `stores/layers/index.ts` 已把地图视口状态纳入统一前端状态：
  - `currentMapCenter`
  - `currentMapBBox`
  - `particleFlowCatalogId`
- 这意味着天气 workflow 的提交参数，已经会显式携带当前地图中心点与可见范围，而不是只依赖 catalog 默认值。
- 旧组件 `Legend.vue`、`Map2D.vue` 已删除，地图职责已实质收敛到 `MapCanvas.vue`。

### 2.3 后端 workflow 主链

后端核心主链仍然是：

- `api/routes.py`
- `services/interaction_hub.py`
- `tasks/workflow_tasks.py`

但当前事实已发生几个重要变化：

- `interaction_hub.py` 已支持失败分类与自动重试中间态；
- `shared/contracts/api_contracts.py` 已新增：
  - `ExecutionStatus.retry_pending`
  - `FailureCategory`
- `services/bridge_protocol.py` 已新增 `BridgeExecutionError`，bridge 层可以把失败按“可重试 / 不可重试”上抛；
- `services/failure_classifier.py` 已新增，负责异常到 `FailureCategory` 的统一映射。

这意味着 workflow 失败不再统一直接进入 `failed`，而是开始具备自动重试语义。

### 2.4 下载链

下载链不再停留在 `demo://snapshots/...` 占位方案。

当前事实：

- `services/source_fetcher.py` 已新增，支持：

  - `minio://bucket/key`
  - `file:///path`
  - `local://path`
- `services/download_service.py` 已接入：
  - source fetch artifact 持久化
  - `partial_success`

- `core/config.py` 已新增下载抓取相关配置：
  - `download_source_root`
  - `download_source_uri_map`

当前可理解为：下载链已经进入“真实抓取骨架已落地”的阶段。

### 2.5 weatherengine 与 weather workflow

当前天气能力同时存在两条路径：

#### A. fallback weatherengine 路径

由：

- `weatherengine/service.py`
- `weatherengine/client.py`
- `weatherengine/constants.py`

提供。

主要职责：

- `/weather/point` 点天气查询
- fallback `execute()` 输出 `map_layer`
- GeoJSON / COG / `cog_preview_url` / `cog_bbox`
- 供前端直接消费 `render_hint`、`point_feature`、`layer_assets`

#### B. weather workflow DAG 路径

由：

- `services/weather_bridge_service.py`
- `weatherengine/workflow_service.py`
- `weatherengine/nodes/*`

提供。

当前已注册节点包括：

- `ForecastFetchNode`
- `PointParseNode`
- `WindFieldRenderNode`
- `TemperatureGridRenderNode`
- `PrecipitationGridRenderNode`
- `HumidityGridRenderNode`
- `PressureGridRenderNode`
- `VisibilityGridRenderNode`
- `SummaryGenerateNode`

这表示天气工作流不再只覆盖风/温/雨三层，已经扩到湿度、气压、能见度。

补充事实：

- `weatherengine/workflow_service.py` 现在已经是天气节点注册与 DAG 执行的真实入口，而不只是实验性封装；
- `services/weather_bridge_service.py` 负责把 `workflow-runs` 主链路由到天气 DAG；
- fallback `weatherengine/service.py` 仍保留直接产出 `map_layer` 的职责，因此当前天气执行面是 “fallback 产物链 + workflow DAG” 双路径并存。
