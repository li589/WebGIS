# OMEGA 调度资源与运行建议

## 目的

本文档把当前 OMEGA 的实现特征、正式基线和工程约束，转译成 WebGIS 后端调度器可直接采用的运行建议。

本文档回答的是：

1. OMEGA 更像什么类型的任务
2. 调度器初始应给多少资源
3. 哪些参数会明显改变 I/O 和产物数量
4. 哪些 profiling 能力只适合离线，不应进生产主链

## 当前任务形态

从当前代码看，OMEGA 更接近“CPU 密集 + MAT 文件 I/O + 批处理产物写盘”的任务，而不是在线低延迟接口。

主要特征如下：

1. 推荐入口是 `retrieval_workflow -> timeseries_bundle -> omega_block`
2. 运行时会读取和写出多个 MAT 文件
3. 当前没有 GPU 依赖，也没有 GPU 加速路径
4. `write_daily_files=true` 时会额外写出逐日文件，显著增加产物数和磁盘 I/O
5. physics 层存在 `Numba` 可回退装载，但不是运行前提

因此：

1. 调度器应按批处理任务看待 OMEGA
2. 不建议把它当成需要秒级同步响应的 HTTP 即时计算

## 当前正式基线

基于当前已记录的正式验证文档，已有如下离线基线：

### `formal96`

1. `execute_omega_retrieval.avg_ms = 12725.158`
2. `single_pixel_solver.avg_ms_per_pixel = 128.036`

### `dual96`

1. `execute_omega_retrieval.avg_ms = 12067.264`
2. `single_pixel_solver.avg_ms_per_pixel = 124.436`

解释：

1. 这两组结果来自离线正式验证，不是 HTTP SLA 承诺
2. 其价值主要在于提供调度初始超时和资源档位参考
3. 真实生产运行时间还会受到输入规模、磁盘、并发和路径解析方式影响

## 推荐资源档位

### 初始推荐

在没有进一步平台压测前，建议把 OMEGA 作业的默认资源提示设为：

1. `cpu_cores = 8`
2. `memory_gb = 32`
3. `gpu_count = 0`
4. `tmp_disk_gb = 10-20`

说明：

1. `cpu_cores = 8` 和 `memory_gb = 32` 适合作为当前平台默认起点
2. `gpu_count = 0` 是因为当前代码没有 GPU 路径
3. `tmp_disk_gb` 需要覆盖中间 MAT 和 manifest 写盘，尤其是开启日文件输出时
4. 这些值属于调度建议，不改变算法计算语义

### 为什么先用保守资源

当前仓库已经证明：

1. 前向核内核化已落地
2. 若干更深层实验性原型已被正式基线证伪
3. 当前稳定版本更适合先做可控上线，而不是极限压榨资源

因此资源策略也应偏保守，先保证可重复运行，再做平台级压测收紧。

## 推荐超时策略

当前建议不要把超时压得过紧。

初始可采用以下策略：

1. 单个 OMEGA block 作业先给 `15 min` 调度超时
2. 如果平台还包含数据下载、路径挂载、对象存储回传，建议进一步上调
3. 只有在完成平台真机压测后，再按分档策略细化

原因：

1. 当前离线基线主要反映算法主计算阶段
2. 平台真实耗时还包含输入发现、MAT 读写、日志写出、manifest 落盘等环节
3. `write_daily_files=true` 会放大 I/O 方差

## 并发建议

当前不建议对 OMEGA 任务做过高并发堆叠。

推荐策略：

1. 同一 worker 节点上先限制 OMEGA 并发数
2. `write_daily_files=true` 时进一步降低单机并发
3. 若平台采用按区域或时间分片提交，优先做队列级排队，而不是单机硬并发

原因：

1. 任务既有 CPU 计算，也有较重 MAT 写盘
2. 并发过高时，磁盘争用和缓存抖动可能让平均耗时恶化
3. 当前正式基线是单任务验证，不代表高并发下仍保持同一时间分布

## 参数对调度的影响

### `write_daily_files`

这是当前最直接影响产物数量和 I/O 的参数。

当 `write_daily_files=true` 时：

1. 除 `omega_block_mat` 外，还会写出多个 `omega_daily_mat`
2. 输出目录下会创建 `daily_omega` 子目录
3. 调度器需要预留更多磁盘空间和上传时间

建议：

1. 如果平台当前只消费块级结果，可优先设为 `false`
2. 如果平台确实要逐日产品，再开启该参数

### `temp_scheme`

当前这是算法配置项，也会影响运行路径。

建议：

1. 平台初期只开放已验证过的标准取值
2. 不要在上线初期同时开放太多模式切换

### `exp_mode`

当前允许值为：

1. `Exp0`
2. `EXP1A`
3. `EXP1B`
4. `EXP2`

建议：

1. 平台首发只开放默认常用值
2. 其他实验模式通过白名单或运维开关逐步放出

## 推荐的 `resource_hint`

平台提交 OMEGA 请求时，可直接附带如下资源提示：

```json
{
  "resource_hint": {
    "cpu_cores": 8,
    "memory_gb": 32,
    "gpu_count": 0,
    "tmp_disk_gb": 16
  }
}
```

说明：

1. 当前代码会把该对象反序列化为 `ResourceHint`
2. 它更适合作为调度器选机和限流依据
3. 它不是算法参数，不应由业务前端自由改写

## 生产运行建议

### 建议保留的产物

平台至少应稳定保留：

1. `JobResult`
2. `ProductManifest`
3. 结构化日志
4. `omega_block_mat`
5. 若开启逐日输出，则保留 `omega_daily_mat`

### 建议记录的日志字段

平台最好至少记录：

1. `job_id`
2. `run_id`
3. `event_type`
4. `stage`
5. `message`
6. `artifact_uri`

这样才能在运行退化时区分：

1. 算法计算慢
2. 文件 I/O 慢
3. 调度排队慢
4. 上传和持久化慢

### 建议区分在线和离线任务

当前建议把任务分成两类：

1. 生产执行任务
2. 离线诊断任务

其中以下 profile 或分析动作，应只保留在离线诊断链路：

1. `formal96`
2. `dual96`
3. `formal192`
4. `cprofile96`

原因是它们用于回归和归因，不是给调度器在线执行的业务模式。

## 当前不建议做的事

1. 不要把 `cProfile` 或 formal profile 暴露成线上 API 参数
2. 不要在生产调度版本中混入 residual/Jacobian/evaluator 的实验性原型
3. 不要默认开启过高并发来追求吞吐
4. 不要依赖硬编码文件名推断任务是否成功，始终以 manifest 为准

## 平台首发建议

如果当前目标是“先接起来，再逐步放量”，推荐采用以下上线方式：

1. 入口固定为 `workflow_name = retrieval_workflow`
2. `algorithm_params.mode` 固定为 `omega`
3. 默认 `write_daily_files = false`，把日产品作为第二阶段能力
4. 单任务默认资源从 `8C / 32G / 无 GPU` 开始
5. 单 worker 控制并发，先跑稳定性观察
6. manifest、日志、错误响应全部结构化落盘

## 后续建议

完成首轮平台压测后，建议再补以下数据：

1. 不同像元规模下的耗时分档
2. 开启和关闭 `write_daily_files` 的 I/O 差值
3. 不同并发数下的节点资源利用率
4. 不同存储后端下的 MAT 读写方差

只有补齐这些平台侧证据后，才适合进一步收紧资源或扩大并发。
