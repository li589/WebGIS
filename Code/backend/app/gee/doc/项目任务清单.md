# WebGIS GEE 模块项目任务清单

## 1. 当前阶段定义

- 当前阶段：P2 第二阶段
- 当前目标：在已完成 Redis 租约续约、心跳和恢复降级策略基础上，继续推进高压场景验证、后端路由契约收口、后续分析节点扩展，以及 `schema 1.12+` 工作流版本管理演进
- 当前产出：可执行 `core` 代码、更新后的任务清单、阶段工作记录

## 2. 里程碑规划

### M1. 文档与骨架阶段

- 优先级：P0
- 目标：形成可落地的技术方案和项目骨架
- 验收标准：
  - `doc` 下具备完整设计文档
  - `core` 目录结构完成初始化
  - 项目依赖、配置和包结构确定

### M2. 引擎最小可用阶段

- 优先级：P0
- 目标：实现最小可用 GEE 引擎能力
- 验收标准：
  - 可加载工作流定义并校验
  - 可执行最小节点链路
  - 可完成本地或 MinIO 结果落盘
  - 可完成基础诊断

### M3. 多账号与批处理阶段

- 优先级：P1
- 目标：支撑真实作业场景
- 验收标准：
  - 支持多账号租约和冷却切换
  - 支持批量拆分任务
  - 支持导出状态轮询和产物索引

### M4. 平台集成阶段

- 优先级：P1
- 目标：与 WebGIS 上层后端打通
- 验收标准：
  - 可被 FastAPI 封装为标准接口
  - 可在 Celery worker 中稳定运行
  - 具备基础日志与指标

## 4. 建议开发顺序

1. 核心模型

4. 账号池与存储抽象

7. 测试、后端路由与 FastAPI 集成适配

## 5. 核心任务说明

### 5.1 工作流模型

- 目标：定义可序列化、可校验、可扩展的 DAG 结构
- 重点：节点参数模式、端口类型、依赖顺序、运行策略
- 产出：工作流 DTO、领域模型、校验器

### 5.2 账号池

- 目标：支撑多账号并发和超限切换
- 重点：租约、状态机、冷却、回收、故障隔离
- 产出：账号实体、仓储接口、选择策略、诊断结果

### 5.3 存储抽象

- 目标：统一本地和 MinIO 访问方式
- 重点：路径规则、上传下载、产物索引、错误封装
- 产出：存储接口、后端实现、artifact 模型

### 5.4 GEE 运行时

- 目标：把 GEE 能力收束在运行时和适配层中
- 重点：初始化隔离、错误分类、导出轮询、执行封装
- 产出：GEE 上下文、运行器、适配器、诊断器

## 6. 风险清单

- 风险 1：GEE Python API 初始化存在全局状态，容易导致多账号串上下文
- 风险 2：大文件导出与上传链路可能造成 worker 内存峰值过高
- 风险 3：节点类型如果前期不收敛，后续工作流扩展会失控
- 风险 4：本地与 MinIO 若没有统一 artifact 模型，后续接入平台会变复杂
- 风险 5：Celery 并发策略与账号池策略不匹配时，可能出现资源泄漏
- 风险 6：上层若绕过 `manifest_uri` 可信映射和权限校验，可能导致跨任务或跨租户查询；当前 `core` 已限制存储根目录与 `exports/` 前缀，上层仍需校验业务归属

## 7. 近期执行建议

建议下一轮继续推进 `P2`，具体包括：

1. 继续补多账号、多批次和长轮询组合压力场景
2. 继续完善工作流版本管理，优先补更多有证据支撑的真实节点兼容规则与 `1.12+` schema 演进策略
3. 补共享 Redis 配额在跨 worker 竞争、续约失败、恢复窗口和异常演化下的压力场景
4. 视上层技术栈补充 Prometheus / OpenTelemetry / ELK 的具体适配器实现
5. 继续细化版本诊断输出与兼容提示
6. 将 `create_api_router()` 实际挂入 WebGIS 后端，并补认证、中间件和统一错误模型适配
7. 在 WebGIS 上层落地 `biz_job_id/run_id/manifest_uri` 可信映射，轮询写回时显式传 `update_manifest=True`
