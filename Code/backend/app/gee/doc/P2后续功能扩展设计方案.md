# P2 后续功能扩展设计方案

## 1. 目标

1. 在当前资源治理、共享 Redis 配额和异常恢复边界已经稳定的前提下，继续扩展 GEE 核心能力。
2. 将下一阶段工作收敛为两条主线：
   - 分析节点扩展
   - 工作流版本管理
3. 保持扩展设计不破坏现有 `WorkflowService`、节点注册体系、诊断能力和薄接口契约。

## 2. 设计原则

1. 先保证核心稳定性，再追加功能复杂度。
2. 新能力优先复用现有节点规范、错误模型、资源治理与可观测性埋点。
3. 工作流版本管理必须服务于长期兼容，不只做字段占位。
4. 分析节点优先覆盖高频遥感处理能力，不做低价值堆砌。

## 3. 当前基线

当前 `core` 已具备以下基础：

1. 工作流校验、拓扑执行、导出归档和轮询回写闭环。
2. 多账号池、错误分类联动、运行时资源控制、共享 Redis 配额协调。
3. Redis 共享租约续约、恢复降级策略，以及跨 worker 恢复窗口共享能力。
4. 第一批与第二批分析节点：
   - 波段选择
   - 光谱指数
   - 影像集合合成
   - 区域统计
   - 时序统计

## 4. 功能扩展顺序

建议按以下顺序推进：

1. 第三批分析节点
2. 第四批分析节点或分析辅助节点
3. 工作流版本模型增强
4. 工作流迁移与兼容校验
5. 版本诊断与灰度接入

原因：

1. 分析节点扩展能直接提升 GEE 可用能力面。
2. 工作流版本管理会影响持久化、平台集成和长期兼容，适合在节点集继续扩展前尽快定型。

## 5. 分析节点扩展设计

### 5.1 第三批分析节点

优先补以下三类：

1. `gee_raster_algebra`
2. `gee_threshold_classify`
3. `gee_reclassify`

### 5.2 gee_raster_algebra

目标：

1. 支持对单影像或多波段影像执行基础栅格代数表达式。
2. 统一承接更复杂的指数推导、条件运算和组合计算。

建议输入：

1. `image`
2. `expression`
3. `band_map`
4. `output_band`

建议输出：

1. `image`

最小实现策略：

1. 第一版优先支持基于 GEE `expression()` 的表达式执行。
2. `band_map` 用于把表达式变量映射到实际波段。
3. 对 `expression`、`band_map` 和 `output_band` 做严格校验。

关键校验：

1. `expression` 必须是非空字符串。
2. `band_map` 必须是非空字典，值为非空波段名。
3. `output_band` 必须是非空字符串。

### 5.3 gee_threshold_classify

目标：

1. 根据一个或多个阈值将连续栅格快速分成离散类别。
2. 满足常见的干旱等级、植被等级、水体等级等场景。

建议输入：

1. `image`
2. `band`
3. `thresholds`
4. `class_values`
5. `output_band`

建议输出：

1. `image`

最小实现策略：

1. 第一版支持升序阈值。
2. 类别值数量固定为 `len(thresholds) + 1`。
3. 采用链式条件表达式构建分类结果。

关键校验：

1. `thresholds` 必须为升序数值列表。
2. `class_values` 长度必须与阈值规则一致。
3. `band` 与 `output_band` 必须为非空字符串。

### 5.4 gee_reclassify

目标：

1. 支持按区间或精确值重分类已有离散栅格。
2. 便于土地利用编码映射、风险等级归并和专题图重编码。

建议输入：

1. `image`
2. `band`
3. `rules`
4. `default_value`
5. `output_band`

建议输出：

1. `image`

规则结构建议：

1. 精确值映射：
   - `{"match": 10, "value": 1}`
2. 区间映射：
   - `{"min": 0, "max": 0.2, "value": 1}`

最小实现策略：

1. 第一版优先支持精确值匹配与闭区间映射。
2. 内部统一转换为顺序条件判断。

### 5.5 分析辅助节点

第三批落地后，可继续补以下辅助节点：

1. `gee_mask_by_threshold`
2. `gee_band_math_constants`
3. `gee_reduce_neighborhood`

其中优先级建议：

1. `gee_mask_by_threshold`
2. `gee_reduce_neighborhood`
3. `gee_band_math_constants`

### 5.6 分析节点通用规范

所有新增分析节点都应统一遵循：

1. 使用 `NodeSpec` 明确定义输入输出端口。
2. 失败时返回 `RunStatus.FAILED`，不要直接抛裸异常。
3. 在节点执行成功/失败时继续走现有结构化事件和指标埋点。
4. 补齐成功与失败路径单测。
5. 至少补一条与导出节点串联的集成测试。

## 6. 工作流版本管理设计

### 6.1 目标

1. 让工作流定义能够长期演化且可被稳定读取。
2. 为未来节点参数变更、节点重命名和默认值迁移提供基础设施。
3. 避免上层 WebGIS 长期持久化的工作流在升级后无法运行。

### 6.2 版本层次

建议引入两层版本：

1. 工作流版本：
   - `workflow.version`
2. 节点版本：
   - `node.version`

同时新增一个概念：

1. `schema_version`

作用区分：

1. `schema_version` 表示工作流定义结构版本。
2. `workflow.version` 表示该工作流业务版本。
3. `node.version` 表示节点实现/参数契约版本。

### 6.3 建议模型增强

建议在 `WorkflowDefinition.metadata` 或显式字段中补充：

1. `schema_version`
2. `created_at`
3. `updated_at`
4. `source`
5. `compatibility_mode`

建议在节点层增加可选扩展元数据：

1. `deprecated`
2. `replacement_node_type`
3. `parameter_aliases`

### 6.4 迁移机制

建议新增一层迁移器：

1. `WorkflowDefinitionMigrator`

核心职责：

1. 将旧 `schema_version` 工作流升级到当前版本。
2. 对旧节点参数做默认值补齐、字段改名和兼容转换。
3. 在无法自动迁移时返回明确失败原因。

建议能力：

1. `detect_version(workflow_payload)`
2. `migrate_to_latest(workflow_payload)`
3. `validate_compatibility(workflow_definition)`

### 6.5 兼容策略

建议采用“向前迁移、运行时只执行当前版本”的策略：

1. 接收旧版本工作流时，先迁移到当前版本。
2. 执行器和校验器只面对当前版本领域模型。
3. 对废弃节点提供一段过渡期映射表。

### 6.6 校验与诊断

建议新增以下校验规则：

1. `schema_version` 是否受支持。
2. 节点 `node_type + version` 是否仍可执行。
3. 是否命中过期节点或过期参数。
4. 是否发生了自动迁移。

建议在 `diagnose()` 或专门工具中暴露：

1. 当前支持的 `schema_version`
2. 已废弃节点列表
3. 可自动迁移的版本范围

## 7. 建议测试策略

### 7.1 分析节点

每个新节点至少补：

1. 参数合法成功路径
2. 参数非法失败路径
3. 输入缺失失败路径
4. 与导出节点串联集成路径

### 7.2 工作流版本管理

至少补：

1. 当前版本工作流正常执行
2. 旧版本工作流自动迁移成功
3. 不可迁移工作流明确失败
4. 废弃节点替换提示或映射成功

## 8. 建议落地顺序

### 第一阶段

1. 落地 `gee_raster_algebra`
2. 落地 `gee_threshold_classify`
3. 补对应单测与导出串联集成测试

### 第二阶段

1. 落地 `gee_reclassify`
2. 补分析辅助节点第一版
3. 收敛节点参数规范和错误信息格式

### 第三阶段

1. 引入 `schema_version`
2. 增加工作流迁移器骨架
3. 为已存在节点补版本兼容元数据

### 第四阶段

1. 完善迁移规则
2. 增加版本诊断输出
3. 对接上层持久化与灰度发布需求

## 9. 结论

建议下一阶段按以下主线执行：

1. 先扩展第三批分析节点，直接提升 GEE 工作流表达能力。
2. 再落地工作流版本管理骨架，为后续节点持续演进打底。
3. 所有新增功能都继续复用现有资源治理、结构化事件、指标和诊断体系，避免功能增长掩盖稳定性问题。
