# WebGIS GEE 模块详细设计方案

## 1. 模块定位

本项目作为 WebGIS 后端体系中的 `gee` 子模块存在，负责提供可复用、可编排、可诊断的 GEE 基础能力，不承担平台级调度器职责。上层调度仍由 WebGIS 现有的 `FastAPI + Celery + Redis` 作业架构负责。

本模块目标不是实现一个面向最终用户的地图应用，而是实现一层标准化 GEE 引擎能力层，为 WebGIS 其他业务模块提供统一调用入口。

## 2. 设计目标

### 2.1 核心目标

1. 提供类似 ComfyUI 的节点化工作流能力，但仅聚焦 GEE 能力编排。
2. 支持多 GEE 账号池调度、租约管理、超限回退和自动切换。
3. 支持批量地理数据处理、导出、结果落盘和统一结果封装。
4. 对外暴露统一 Python Service 接口，便于后续被 FastAPI 直接包装。
5. 支持本地文件系统和 MinIO 双存储后端，保持存储解耦。
6. 提供连接诊断、账号健康检查、运行指标和错误分类能力。

### 2.2 非目标

1. 不在本模块内实现平台级任务调度器。
2. 不直接承载前端工作流画布页面。
3. 第一阶段不追求覆盖所有遥感算法，只先实现底层通用节点。
4. 不把复杂的大规模数据缓存长期放在进程内存中。

## 3. 模块边界

### 3.1 WebGIS 平台负责

1. 用户鉴权、租户体系、权限控制。
2. HTTP 接口暴露、作业创建、作业优先级管理。
3. Celery 调度、任务队列、运行态持久化。
4. 业务级流程编排和页面交互。

### 3.2 GEE 模块负责

1. 工作流定义校验与节点执行。
2. GEE 客户端初始化和执行上下文隔离。
3. 多账号池管理、租约申请、冷却与回收。
4. GEE 导出接收、产物整理和存储落盘。
5. 统一结果对象、诊断结果和异常分类。

## 4. 目录约定

仓库内部统一采用如下约定：

```text
gee/
├─ core/                       # 所有 Python 代码
└─ doc/                        # 所有设计文档、任务清单、记录文档
```

建议在 `core/` 内采用如下结构。当前已实际落地的是主干目录，`artifacts/`、`schemas/`、`utils/` 等扩展目录后续按需要再补充：

```text
core/
├─ pyproject.toml
├─ README.md
├─ src/
│  └─ webgis_gee/
│     ├─ api/                  # 对外服务接口
│     ├─ application/          # 应用服务编排层
│     ├─ domain/               # 核心实体、值对象、领域接口
│     ├─ workflow/             # DAG、工作流校验、执行计划
│     ├─ nodes/                # 节点定义、注册器、节点实现
│     ├─ runtime/              # 执行上下文、运行器、重试机制
│     ├─ accounts/             # 账号池、租约、健康检查、限额控制
│     ├─ gee/                  # GEE 客户端适配层
│     ├─ storage/              # 本地与 MinIO 存储后端
│     ├─ artifacts/            # 可选扩展：产物索引、manifest、导出记录
│     ├─ diagnostics/          # 连接诊断、账号诊断、运行诊断
│     ├─ schemas/              # 可选扩展：DTO、Pydantic 模型

│     └─ utils/                # 可选扩展：通用工具函数
├─ tests/
│  ├─ unit/
│  ├─ integration/
│  └─ fixtures/                # 可选扩展：测试夹具
└─ scripts/                    # 可选扩展：脚本工具
```

## 5. 分层架构设计

### 5.1 架构分层

1. `API Facade`
   - 暴露统一入口，例如 `execute_workflow()`、`validate_workflow()`、`diagnose()`
   - 只负责参数接入与结果输出，不承担复杂业务逻辑
2. `Application Layer`
   - 负责工作流执行编排、节点调用顺序、结果聚合
   - 将上层请求转成领域对象和运行命令
3. `Domain Layer`
   - 定义工作流、节点、账号租约、产物、诊断报告等核心模型
   - 定义抽象接口，例如存储接口、账号仓储接口、执行器接口
4. `Infrastructure Layer`
   - 对接 GEE、MinIO、本地文件系统、Redis、日志、监控
   - 放置所有外部依赖适配逻辑

### 5.2 设计原则

1. 节点定义标准化，执行上下文统一注入。
2. GEE 对象尽量惰性传递，不提前拉取本地大数组。
3. 多账号切换由运行时统一处理，不侵入节点业务。
4. 存储能力通过接口抽象，业务层不区分本地和 MinIO。
5. 错误必须分类，支持精细化重试与故障转移。

## 6. 工作流模型设计

### 6.1 工作流对象

工作流建议采用 JSON 可序列化结构，核心包含：

1. `workflow_id`
2. `version`
3. `nodes`
4. `edges`
5. `inputs`
6. `runtime_policy`
7. `storage_policy`
8. `metadata`

### 6.2 节点定义模型

每个节点至少包含以下字段：

1. `node_id`
2. `node_type`
3. `version`
4. `params`
5. `input_ports`
6. `output_ports`
7. `resource_profile`
8. `batch_strategy`
9. `retry_policy`
10. `ui_schema`

### 6.3 端口类型建议

1. `GeometryRef`
2. `FeatureCollectionRef`
3. `ImageRef`
4. `ImageCollectionRef`
5. `TableRef`
6. `RasterArtifact`
7. `VectorArtifact`
8. `ExportTaskRef`
9. `StorageURI`
10. `DiagnosticReport`

### 6.4 工作流执行阶段

1. 参数绑定
2. 结构校验
3. 拓扑排序
4. 账号租约申请
5. 节点逐步执行
6. 导出和产物归档
7. 结果汇总
8. 资源释放和账号归还

## 7. 节点体系设计

### 7.1 第一阶段节点分组

1. 数据输入节点
   - 区域输入
   - 影像集合加载
   - 单景影像加载
   - 资产引用
2. 过滤预处理节点
   - 时间过滤
   - 空间过滤
   - 波段选择
   - 云掩膜
   - 裁剪
   - 重采样
3. 分析计算节点
   - 指数计算
   - 时序统计
   - 区域统计
   - 栅格代数
4. 批处理节点
   - 按区域拆分

5. 导出节点
   - 导出栅格
   - 导出统计表
   - 导出 manifest
6. 运维节点
   - 连接诊断
   - 账号诊断
   - 配额检查
   - 任务轮询

### 7.2 节点实现规范

每个节点实现建议包含：

1. 参数 Pydantic 模型
2. 输入输出端口定义
3. 执行函数
4. 可观测指标埋点
5. 异常映射
6. UI 元信息

## 8. 多账号管理设计

### 8.1 账号状态机

建议定义以下账号状态：

1. `AVAILABLE`
2. `LEASED`
3. `COOLDOWN`
4. `DISABLED`
5. `EXHAUSTED`

### 8.2 核心能力

1. 账号配置加载
2. 账号租约申请与续租
3. 执行完成自动归还
4. 超限后冷却
5. 按健康度选择账号
6. 异常统计与熔断

### 8.3 超限回退策略

1. 瞬时错误：有限重试后继续当前账号。
2. 并发受限：当前账号进入短冷却，任务切换账号重试。
3. 日限额耗尽：账号进入长冷却或禁用状态。
4. 权限错误：标记为配置问题，不自动重试。
5. 参数错误：直接终止当前节点执行。

### 8.4 并发建议

1. 避免多个账号共享一个全局 GEE 初始化上下文。
2. 优先采用任务级或进程级 GEE 上下文隔离。
3. 在 Celery worker 中控制同进程内 GEE 初始化复用边界。

## 9. 存储与产物管理设计

### 9.1 存储抽象接口

统一定义 `StorageBackend` 接口。当前实现已包含：

1. `put`
2. `get`
3. `exists`
4. `delete`
5. `list`
6. `stat`
7. `build_uri`

如后续出现大文件流式读写需求，再补充 `open_reader` / `open_writer` 一类流式接口。

### 9.2 后端实现

1. `LocalStorageBackend`
2. `MinioStorageBackend`

### 9.3 产物模型

建议定义 `ArtifactRecord`：

1. `artifact_id`
2. `workflow_run_id`
3. `node_id`
4. `artifact_type`
5. `storage_uri`
6. `content_type`
7. `size`
8. `checksum`
9. `created_at`
10. `metadata`

### 9.4 路径规划

建议产物路径统一按如下逻辑组织：

```text
{tenant}/{project}/{workflow_run_id}/{node_id}/{partition_key}/{filename}
```

## 10. 对外接口设计

### 10.1 核心服务接口

当前建议区分“内部服务层接口”和“WebGIS 上层适配契约”两层：

内部服务层接口：

1. `validate_workflow(definition)`
2. `execute_workflow(definition, context=None)`
3. `diagnose()`
4. `poll_export_task(manifest_uri, task_ref=None)`

WebGIS 上层适配契约：

1. `submit_workflow(workflow, context)`
2. `get_export_task_status(manifest_uri)`
3. `run_workflow_job(payload)`

### 10.2 返回对象建议

工作流执行统一返回 `RunResult`，当前包含：

1. `run_id`
2. `status`
3. `outputs`
4. `artifacts`
5. `warnings`
6. `errors`

如后续增强可观测性，再扩展 `metrics`、`account_trace` 等字段。

### 10.3 错误分类建议

1. `WorkflowValidationError`
2. `NodeExecutionError`
3. `TransientGEEError`
4. `GEEQuotaExceededError`
5. `AccountUnavailableError`
6. `StorageOperationError`
7. `ExportTimeoutError`
8. `ConfigurationError`

### 11.2 指标建议

3. 账号租约命中率
4. 超限次数
5. 导出成功率

### 11.3 日志建议

1. 统一 `run_id`
2. 统一 `node_id`
3. 统一 `account_id`
4. 结构化日志输出
5. 关键阶段打印开始、结束、耗时和异常原因

## 12. 性能与内存管理设计

1. 尽量在 GEE 侧完成空间计算，本地不提前物化大对象。
2. 大文件中间产物走磁盘临时目录，不走内存长驻缓存。
3. 批任务按分片清单推进，避免单次持有过多上下文。
4. 导出、下载、上传设置并发阈值，防止 worker 内存峰值过高。
5. 对临时目录、失败中间件、孤儿文件设置回收机制。

## 15. 分阶段建设路线

### 15.1 P0 基础骨架

1. 建立 `core` 项目结构和包管理文件。
2. 定义核心领域模型和工作流 JSON 结构。
3. 建立节点注册器和最小工作流执行器。
4. 建立账号池、租约和冷却机制基础模型。
5. 建立本地存储和 MinIO 存储抽象。
6. 实现基础诊断能力。

### 15.2 P1 最小可用版本

1. 实现第一批基础节点。
2. 接入导出、下载、落盘和产物索引。
3. 实现错误分类、重试和超限回退。
4. 打通与 Celery worker 的集成方式。
5. 输出可供 FastAPI 直接调用的服务接口。

### 15.3 P2 增强版本

1. 补齐批处理映射节点。
2. 增强监控、指标和日志审计。
3. 补充工作流版本兼容机制。
4. 补充更丰富的遥感分析节点。

## 16. 当前建议落地顺序

当前最合理的执行顺序如下：

1. 先完成文档、目录和核心领域建模。
2. 再完成工作流模型与节点注册体系。
3. 然后实现账号池、存储后端和执行上下文。
4. 最后接入具体节点、导出链路和 WebGIS 上层调用接口。

以上设计保证本模块在后续被整体迁移到 WebGIS 后端 `gee` 目录时，仍然保持目录清晰、边界清晰、接口稳定。
