# 综合地理数据平台 — 后端架构设计

## 1. 架构概览

综合地理数据平台采用三层架构设计，各层职责清晰、边界明确，通过标准化接口实现层间解耦与灵活替换。

```
┌─────────────────────────────────────────────────────┐
│                    前端 (Vue3 + TypeScript + MapLibre GL)              │
│   可视化编辑器 · 工作流编排 · 数据可视化 · 交互控制                           │
└───────────────────────┬─────────────────────────────────────┘
                        │ REST API / WebSocket
┌───────────────────────▼─────────────────────────────────────┐
│                 后端 (FastAPI + SQLite + Celery/Redis)                       │
│   API 路由 · 任务调度 · 存储抽象 · 工作流服务 · 瓦片 / artifact 数据面         │
└───────────────────────┬─────────────────────────────────────┘
                        │ Python Bridge
┌───────────────────────▼─────────────────────────────────────┐
│              Python 算法包 (模块化工作流算法 + Celery 调度)                 │
│   算法模块 · 格式转换 · 数据访问层 · 调度适配器 · 产物输出                    │
└─────────────────────────────────────────────────────┘
```

- **前端层**：基于 Vue3 + TypeScript 构建单页应用，以 MapLibre GL 为地理数据渲染引擎，负责工作流可视化编排、数据可视化与用户交互。
- **后端层**：基于 FastAPI 构建 RESTful API 服务，以 SQLite 持久化工作流定义与任务状态，引入 Celery + Redis 实现任务队列与异步调度。
- **Python 算法包层**：核心算法与数据处理的承载层，以模块化为设计原则，支持本地文件系统与 MinIO 对象存储两种部署模式。

## 2. 核心设计约束

三条核心约束贯穿整个系统架构，定义了数据、格式与编排的根本决策。

### 2.1 数据与项目解耦

数据存储与代码逻辑完全分离，Python 算法包不直接依赖任何特定存储介质。系统通过统一的数据访问层（DataAccessLayer）抽象底层存储细节，算法模块仅消费 `PreparedInput`（统一格式：本地路径字符串或 S3 URI），对数据来源无感知。

- **本地文件系统模式**：适用于导师演示、代码调试及本地开发场景。数据存放于本地目录（如 `D:\GeoData`），产物输出至本地目录（如 `D:\GeoOutput`）。
- **MinIO 对象存储模式**：适用于超算或服务器生产部署。数据存放于 MinIO S3 兼容对象存储桶，产物写入指定桶路径。

### 2.2 MAT 格式保留与转换

`.mat` 格式（MATLAB 专用格式）是课题组的计算枢纽：课题组在 MATLAB 环境中完成核心算法研发与验证，输出 `.mat` 中间结果。Python 算法包承担格式转换职责，将 `.mat` / `.hdf` / `.hdf5` / `.nc` 等原始格式转换为前端可直接消费的标准格式（COG 栅格、PNG 预览图、Parquet 表格、JSON manifest）。

```
原始数据 (.mat / .hdf / .hdf5 / .nc)
        ↓ (Python 算法包 · 格式转换)
┌─ COG 栅格 ────→ 前端 MapLibre GL 直接渲染
├─ PNG 预览图 ──→ 前端缩略图快速展示
├─ Parquet ─────→ 前端表格图表消费
└─ JSON manifest → 前端 InfoPanel 详情展示
```

### 2.3 ComfyUI 式工作流编排

参考 ComfyUI 的节点图执行模型，将算法功能封装为独立的原子单元（模块），前端通过可视化编辑器以拖拽、连线的方式拼装 DAG 工作流，后端调度系统负责 DAG 验证与拓扑顺序执行。

- **模块即节点**：每个算法模块定义明确的输入端口、输出端口与可调参数。
- **DAG 即工作流**：前端编辑器生成的节点图（nodes + edges）作为工作流定义（WorkflowDefinition）提交给后端。
- **调度即执行**：后端按拓扑顺序消费 DAG，节点输出通过 artifact_store 在节点间传递。

## 3. 存储层设计

### 3.1 存储抽象架构

数据访问层（DataAccessLayer）位于 Python 算法包与底层存储介质之间，负责屏蔽存储细节差异，为上层算法模块提供统一的数据访问接口。

```
前端 / 算法模块
    ↓ DataRequest
    │  (dataset_name / variables / time_range / region)
    │
    ▼
┌──────────────────────────────────────────────────────────┐
│              数据访问层（DataAccessLayer）                │
│  ┌────────────────────┐    ┌────────────────────┐       │
│  │ LocalFileSystemAdapter │    │    MinIOAdapter      │       │
│  │    （本地调试/导师演示）    │    （超算/服务器生产部署）   │       │
│  └────────────────────┘    └────────────────────┘       │
└──────────────────────────────────────────────────────────┘
    │
    ▼
PreparedInput（统一格式：本地路径 or S3 URI）
    │
    ▼
算法模块（只消费 PreparedInput，对数据来源无感知）
```

### 3.2 环境变量配置

存储后端的选择与连接参数通过环境变量统一配置：

| 环境变量 | 类型 | 说明 |
|---------|------|------|
| `BACKEND_STORAGE_BACKEND` | `"local"` \| `"minio"` | 存储后端类型，默认为 `"local"` |
| `BACKEND_DATA_ROOT` | 字符串 | 本地数据根目录（如 `D:\GeoData` 或 `/data/geodata`） |
| `BACKEND_OUTPUT_ROOT` | 字符串 | 产物输出根目录（如 `D:\GeoOutput` 或 `/data/geooutput`） |
| `BACKEND_MINIO_ENDPOINT` | 字符串 | MinIO 服务地址（如 `192.168.1.100:9000`） |
| `BACKEND_MINIO_ACCESS_KEY` | 字符串 | MinIO 访问密钥 |
| `BACKEND_MINIO_SECRET_KEY` | 字符串 | MinIO 密钥 |
| `BACKEND_MINIO_BUCKET` | 字符串 | 数据桶名（如 `geodata`） |
| `BACKEND_MINIO_SECURE` | `"true"` \| `"false"` | 是否使用 HTTPS，默认为 `"false"` |

### 3.4 启动时自动检测策略

应用启动时，系统按以下顺序自动选择存储后端：

1. **优先检查本地文件系统**：如果 `BACKEND_DATA_ROOT` 指定的目录存在且可访问，则使用 `LocalFileSystemAdapter`。
2. **降级至 MinIO**：如果本地目录不存在或不可访问，且环境变量中配置了 MinIO 相关参数，则切换至 `MinIOAdapter`。
3. **错误退出**：如果两者均不可用，系统记录错误日志并终止启动，避免以不一致状态运行。

此策略确保本地开发/调试时无需配置 MinIO，同时生产部署时自动使用 MinIO 存储。

## 4. 格式转换层设计

### 4.1 转换职责归属

格式转换的职责完全由 Python 算法包承担，前端仅消费标准化输出。转换链路如下：

```
原始 .mat / .hdf / .hdf5 / .nc 文件
        │
        ▼  Python 算法包 · 格式转换
        │
        ├───────────────────────→ COG 栅格
        │   (前端 MapLibre GL 直接加载，无需二次处理)
        │
        ├───────────────────────→ PNG 预览图
        │   (256x256 缩略图，前端快速预览)
        │
        ├───────────────────────→ Parquet 表格
        │   (前端图表消费)
        │
        └───────────────────────→ JSON manifest
            (前端 InfoPanel 详情展示)
```

### 4.2 publish 写出层职责

Python 包的 `publish/` 目录承担所有格式写出职责，各模块的 `execute()` 方法产出中间结果后，由 publish 层统一完成标准化输出。

| 写出操作 | 使用库 | 说明 |
|---------|--------|------|
| COG 栅格 | `rasterio` + `rio-cogeo` | 含空间参考（CRS/bounds）、NoData 处理、COG 瓦片化 |
| Parquet 表格 | `pandas` + `pyarrow` | 时序数据或统计聚合结果 |
| 缩略图 | `Pillow` | 256x256 PNG，前端快速预览 |
| manifest.json | `json`（Python 内置） | 标准化产物清单 |

**publish 层的设计原则：**
- 每个算法模块的输出先写至临时目录，成功后原子性移动/上传至最终位置。
- 写出失败时清理已产生的部分文件，保持输出目录一致性。
- 支持本地文件系统与 MinIO 两种写出目标。

### 4.3 ComfyUI 视角下的节点类型

格式转换本身也是一种节点类型，遵循与其他算法模块完全相同的接口规范。转换节点可被可视化为工作流中的一个独立节点，前端编辑器可直接拖入连线。

```
[读取 .mat] → [提取变量] → [算法处理] → [输出 COG] → [生成预览图] → [组装 manifest]
        │         │           │           │             │              │
      Input    Transform    Compute     Transform     Transform      Output
       Node       Node        Node        Node          Node           Node
```

每个节点类型均定义：
- 输入端口（name / data_type / required）
- 输出端口（name / data_type）
- 可调参数（name / type / default / description / 取值范围）
- 执行逻辑（`BaseModule.execute()` 实现）

## 5. 工作流编排层设计

### 5.1 模块作为原子节点

算法模块是系统的核心原子计算单元。每个模块以 `BaseModule` 为基类，通过 `ModuleRegistry` 统一注册管理。模块必须提供完整的 UI 元数据（`ui_metadata`），供前端可视化编辑器渲染配置面板。

**模块元数据示例（伪代码）：**

```python
class NDVIRetrievalModule(BaseModule):
    name = "ndvi_retrieval"
    version = "1.0.0"

    inputs = [
        Port(name="ndvi_file", data_type="file", required=True),
        Port(name="region", data_type="geojson", required=True),
    ]

    outputs = [
        Port(name="result_cog", data_type="file"),
        Port(name="statistics", data_type="dataframe"),
    ]

    params = [
        Param(name="aggregation", type="string", default="monthly",
              description="时间聚合方式", options=["daily", "monthly", "yearly"]),
        Param(name="quality_flag", type="int", default=0,
              description="质量标志位过滤", options=[0, 1, 2, 3]),
    ]

    ui_metadata = {
        "category": "数据获取",
        "display_name": "NDVI 数据获取",
        "icon": "satellite",
        "panel_schema": {...},  # 前端表单配置
    }
```

### 5.2 工作流定义模型

工作流以 DAG（有向无环图）形式定义，包含节点规格、边规格和输出规格：

```python
class WorkflowDefinition:
    workflow_id: str           # 工作流唯一标识（UUID）
    name: str                  # 工作流名称
    description: str           # 工作流描述
    version: int               # 版本号（每次提交递增）

    nodes: list[WorkflowNodeSpec]  # 节点列表
    """
    WorkflowNodeSpec:
      - node_id: str           # 节点唯一标识
      - module_name: str       # 关联的模块名（对应 ModuleRegistry 中的注册名）
      - params: dict           # 用户配置的可调参数
      - input_bindings: dict   # 端口连线映射 {port_name: "from_node_id.from_port_name"}
    """

    edges: list[WorkflowEdge]  # 边列表
    """
    WorkflowEdge:
      - from_node: str         # 源节点 ID
      - from_port: str         # 源端口名
      - to_node: str           # 目标节点 ID
      - to_port: str           # 目标端口名
    """

    outputs: list[WorkflowOutputSpec]  # 输出规格
    """
    WorkflowOutputSpec:
      - name: str              # 输出产物名称
      - node_id: str           # 来源节点 ID
      - port_name: str         # 来源端口名
    """
```

**工作流定义示例（JSON 格式）：**

```json
{
  "workflow_id": "wf-001",
  "name": "NDVI 月度合成",
  "description": "从 VIIRS NDVI 原始数据提取指定区域并计算月度均值",
  "version": 1,
  "nodes": [
    {
      "node_id": "node_001",
      "module_name": "ndvi_retrieval",
      "params": {
        "aggregation": "monthly",
        "quality_flag": 0
      },
      "input_bindings": {
        "ndvi_file": null,
        "region": null
      }
    },
    {
      "node_id": "node_002",
      "module_name": "cog_publisher",
      "params": {},
      "input_bindings": {
        "raster": "node_001.result_cog",
        "metadata": "node_001.statistics"
      }
    }
  ],
  "edges": [
    {"from_node": "node_001", "from_port": "result_cog", "to_node": "node_002", "to_port": "raster"},
    {"from_node": "node_001", "from_port": "statistics", "to_node": "node_002", "to_port": "metadata"}
  ],
  "outputs": [
    {"name": "ndvi_monthly_cog", "node_id": "node_002", "port_name": "cog_output"},
    {"name": "manifest", "node_id": "node_002", "port_name": "manifest"}
  ]
}
```

### 5.3 三种工作流来源

| 来源 | 说明 | 管理方式 |
|------|------|---------|
| **预设工作流** | 系统内置，如 `retrieval_workflow`，硬编码在 `Python/workflow/presets.py` | 不可修改，与代码库同步 |
| **模板工作流** | 管理员预定义，提供完整配置模板，前端可直接加载并提交 | 数据库管理，管理员可编辑 |
| **用户自建工作流** | 通过 ComfyUI 式可视化编辑器拼装，保存至数据库 | 用户自主管理 |

### 5.4 调度执行流程

```
┌─────────────────────────────────────────────────────────────┐
│                    API 接收 WorkflowDefinition               │
│                  (workflow_id / nodes / edges / outputs)    │
└──────────────────────────┬──────────────────────────────────┘
                           │
                           ▼
              InteractionHub 验证并持久化
                           │
                           ▼
              WorkflowDispatcher 选择执行路径
                           │
              ┌────────────┴────────────┐
              │  sync（同步）  │  celery（异步）  │
              └────────────┬────────────┘
                           │
                           ▼
           Worker 构建 NodeExecutionContext
                           │
                           ▼
           WorkflowRunner 按拓扑顺序执行 DAG
                           │
              ┌────────────┴────────────┐
              │                        │
              ▼                        ▼
    ModuleNodeExecutor              各节点并行
         │                          执行（如无依赖）
         │                           │
         └────────────┬──────────────┘
                      │
                      ▼
         BaseModule.execute() 逐一执行节点
                      │
                      ▼
    节点输出通过 artifact_store 暂存
         (node:node_id.port → artifact)
                      │
                      ▼
           完成后调用 publish/ 写出
           COG + manifest + 预览图 + parquet
                      │
                      ▼
           WorkflowResultReference 返回前端
           (job_id / product_paths / status)
```

### 5.5 关键设计点

**DAG 验证：**
- 依赖关系检查：确保图中无循环依赖，使用 Kahn 算法验证拓扑顺序存在性。
- 端口类型兼容性：连线时检查 from_port 和 to_port 的 data_type 是否兼容。
- 必填输入检查：所有 required=true 的输入端口均已绑定。
- 版本固化：工作流提交时自动递增 version 号，并保存快照。

**Artifact 节点间传递：**
- 节点输出以 artifact 形式暂存在 artifact_store 中（本地文件或 MinIO）。
- 引用语法：`node:{node_id}.{port_name}`，例如 `node:node_001.result_cog`。
- 调度器在执行节点前，自动解析所有 `input_bindings`，从 artifact_store 中获取前置节点的输出。

**动态参数注入：**
- 前端编辑器配置的参数通过 `workflow.nodes[].params` 传入。
- 每个节点的 `ModuleNodeExecutor` 在调用 `BaseModule.execute()` 前，将 `params` 与模块默认参数合并（用户参数覆盖默认参数）。

**工作流版本管理：**
- 每次工作流提交均创建新的 version 号。
- 历史版本不可修改，仅可查询。
- 运行记录（workflow_run）绑定至特定版本号，确保可复现性。

## 7. 实施路线

### 阶段 1（近期）— 基础打通

**目标**：构建完整的数据流转闭环，验证端到端通路。

- 完成存储抽象层（`DataAccessLayer`）的双后端实现（LocalFileSystemAdapter / MinIOAdapter）
- 完成 `publish/` 写出层（COG + Parquet + 预览图 + manifest）
- 配置 VIIRS NDVI 真实数据（`.hdf` 格式），跑通从数据读取 → 算法处理 → COG 输出 → 前端渲染的全链路
- 验证 LocalFileSystemAdapter 在 Windows 本地的读写正确性

### 阶段 2（中期）— 前后端对接

**目标**：将 Python 算法包完整接入前端，形成可用的原型系统。

- 补齐所有算法模块的 UI 元数据（`ui_metadata` / `panel_schema`），支持前端动态渲染配置面板
- 前端接入真实 `workflow-runs` API（创建任务、查询状态、获取产物）
- 接通 Celery Worker 真实链路，实现异步任务调度
- 实现工作流版本管理和历史运行记录查询

### 阶段 3（远期）— ComfyUI 阶段

**目标**：实现完整的可视化工作流编排能力。

- 前端实现 ComfyUI 式可视化编辑器（拖拽模块节点、节点连线、参数配置面板）
- 后端支持用户自建工作流提交（DAG 验证、artifact 链路解析、错误回溯）
- 调度器支持按资源 profile（CPU / 内存 / GPU）分配 worker
- 产物预览能力：在线查看 COG 缩略图、表格分页、JSON 格式化
- 超算作业调度器对接（Slurm / PBS 适配器实现）

## 9. 关键文件索引

### 数据源与存储接口

| 文件路径 | 职责 |

| 文件路径 | 职责 |
|---------|------|

### 算法模块

| 文件路径 | 职责 |
