# 前后端接口与字段规范文档

## 1. 文档目标

本规范用于统一当前 Web GIS 平台中前端展示层、后端接口层、后续算法层之间的字段命名、对象结构与接口语义。

当前第一版规范聚焦三类对象：

- 图层目录对象
- 任务提交对象
- 任务状态对象

同时，第一版规范直接服务于以下接口：

- `GET /layers`
- `POST /tasks`
- `GET /tasks/{task_id}`

## 2. 设计原则

### 2.1 命名原则

- HTTP JSON 字段统一使用 `snake_case`
- 标识字段统一使用 `*_id` 或 `*_key`
- 时间字段统一使用 ISO 8601 时间字符串
- 空间范围统一使用 `bbox + crs`
- 枚举值统一使用稳定的小写英文值

### 2.2 架构原则

- 前端不直接猜测算法字段，而是读取后端返回的图层与任务协议
- 后端不直接暴露算法内部结构，而是通过统一任务协议封装
- `shared/contracts` 作为前后端共享协议的单一事实来源
- 当前任务状态先用内存骨架实现，后续替换为 `Celery + Redis`

## 3. 核心对象模型

### 3.1 图层目录对象 `LayerDescriptor`

用于描述前端图层侧边栏、地图图层控制、模式切换与时间轴可用性。

字段约定：

- `layer_id`: 图层唯一标识，后端稳定主键
- `dataset_key`: 前后端共享的数据集键，用于前端状态与后端任务提交关联
- `display_name`: 面向用户展示的中文图层名称
- `description`: 图层说明
- `category`: 图层类别，如 `meteorology`、`remote_sensing`、`model_output`
- `source_type`: 数据来源类型，如 `demo`、`gee`、`cog`、`algorithm_output`
- `render_type`: 渲染类型，如 `raster`、`vector`、`point`、`heatmap`
- `supported_map_modes`: 支持的地图模式，取值为 `2d` 或 `3d`
- `supports_time`: 是否支持时间轴
- `is_realtime`: 是否属于实时图层
- `default_visible`: 是否默认显示
- `status`: 当前图层状态，第一版取 `available`
- `time_granularity`: 时间粒度，如 `hour`、`day`、`month`
- `default_time_offset`: 默认时间偏移量，便于前端初始化
- `extent`: 图层可用空间范围
- `style`: 前端样式提示，如色带、单位、透明度
- `tags`: 标签列表

### 3.2 空间过滤对象 `SpatialFilter`

用于任务提交时表达研究区域。

字段约定：

- `filter_type`: 第一版默认 `bbox`
- `bbox`: 边界框对象
- `region_code`: 可选区域编码
- `region_name`: 可选区域名称

### 3.3 时间范围对象 `TimeRange`

用于统一任务提交与状态查询中的时间维度。

字段约定：

- `start_at`: 开始时间
- `end_at`: 结束时间
- `granularity`: 时间粒度

### 3.4 任务提交对象 `TaskSubmitRequest`

用于前端在选定图层、时间范围、空间范围后向后端发起计算请求。

字段约定：

- `layer_id`: 目标图层 ID
- `task_type`: 任务类型，第一版支持 `layer_preview`、`analysis`、`export`
- `map_mode`: 当前地图模式，支持 `2d`、`3d`
- `spatial_filter`: 空间过滤条件
- `time_range`: 时间范围
- `parameters`: 自定义算法参数
- `requested_outputs`: 请求输出类型，如 `json`、`tile_url`、`geojson`
- `client_context`: 前端上下文信息，如页面来源、用户交互状态

### 3.5 任务状态对象 `TaskStatusResponse`

用于轮询任务进度并返回结果引用。

字段约定：

- `task_id`: 任务 ID
- `layer_id`: 关联图层 ID
- `task_type`: 任务类型
- `status`: 任务状态，第一版支持 `queued`、`running`、`succeeded`、`failed`
- `progress`: 进度百分比，0 到 100
- `message`: 当前状态说明
- `created_at`: 任务创建时间
- `updated_at`: 最近更新时间
- `spatial_filter`: 提交时使用的空间范围
- `time_range`: 提交时使用的时间范围
- `requested_outputs`: 请求输出列表
- `result_refs`: 结果引用列表
- `diagnostics`: 诊断信息列表

## 4. 字段标准化建议

### 4.1 前端现有字段与标准字段映射

当前前端原型里主要存在以下本地字段：

- `activeDataset`
- `currentHour`
- `mapMode`

建议逐步映射为：

- `activeDataset` -> `layer_id` 或 `dataset_key`
- `currentHour` -> `time_range.start_at / end_at` 的简化 UI 映射
- `mapMode` -> `map_mode`

当前实现阶段补充说明：

- 当前前端主路径已收口为 `2D-first demo`，真实三维地球模式暂未启用
- `map_mode` 在接口协议中继续保留，用于后续对接真实 `2d / 3d` 能力
- 当前时间轴仍以 `currentHour` 作为演示态映射字段，后续接接口时应优先映射到 `time_range`
- 当前图层联动优先验证字段别名映射、可用性状态与时间联动，不代表最终接口字段已经固定

### 4.2 图层展示字段规范

前端图层侧边栏至少应消费这些字段：

- `layer_id`
- `display_name`
- `category`
- `default_visible`
- `supports_time`
- `supported_map_modes`
- `status`

### 4.3 任务表单字段规范

前端任务提交面板至少应生成这些字段：

- `layer_id`
- `map_mode`
- `spatial_filter`
- `time_range`
- `task_type`
- `parameters`

## 5. 当前接口规范

### 5.1 `GET /layers`

用途：

- 获取图层目录
- 驱动前端图层列表、图层说明、模式过滤与时间轴开关

响应结构：

```json
{
  "items": [
    {
      "layer_id": "wind_field_surface",
      "dataset_key": "wind_field",
      "display_name": "风场",
      "description": "演示用近地表风场图层，支持时间轴切换和二维展示。",
      "category": "meteorology",
      "source_type": "demo",
      "render_type": "vector",
      "supported_map_modes": ["2d", "3d"],
      "supports_time": true,
      "is_realtime": true,
      "default_visible": true,
      "status": "available",
      "time_granularity": "hour",
      "default_time_offset": 0,
      "extent": {
        "west": 109.6,
        "south": 20.1,
        "east": 117.4,
        "north": 25.6,
        "crs": "EPSG:4326"
      },
      "style": {
        "palette": "blue-cyan",
        "unit_label": "m/s",
        "opacity": 0.85
      },
      "tags": ["wind", "realtime", "demo"]
    }
  ]
}
```

### 5.2 `POST /tasks`

用途：

- 提交计算任务
- 后续平滑扩展到 `Celery + Redis`

请求示例：

```json
{
  "layer_id": "topic_group_model_output",
  "task_type": "analysis",
  "map_mode": "2d",
  "spatial_filter": {
    "filter_type": "bbox",
    "bbox": {
      "west": 112.9,
      "south": 22.2,
      "east": 114.7,
      "north": 23.9,
      "crs": "EPSG:4326"
    },
    "region_code": "GD-GZ",
    "region_name": "广州"
  },
  "time_range": {
    "start_at": "2026-06-01T00:00:00Z",
    "end_at": "2026-06-02T00:00:00Z",
    "granularity": "day"
  },
  "parameters": {
    "scenario": "demo",
    "threshold": 0.75
  },
  "requested_outputs": ["json"],
  "client_context": {
    "page": "dashboard"
  }
}
```

响应示例：

```json
{
  "task_id": "task-123456789abc",
  "status": "queued",
  "status_url": "/tasks/task-123456789abc",
  "created_at": "2026-06-30T12:00:00Z",
  "message": "任务已提交，可通过状态接口轮询进度。"
}
```

### 5.3 `GET /tasks/{task_id}`

用途：

- 查询任务执行状态
- 为前端轮询与结果区面板提供统一数据源

响应对象包含：

- 任务基本信息
- 进度与消息
- 提交时的时间/空间条件
- 结果引用
- 诊断信息

## 6. 前后端链接建议

### 6.1 前端图层侧边栏

前端的 `LayerSidebar` 不应继续只使用字符串数组，而应改为消费 `GET /layers` 返回的 `items`。

推荐最小渲染字段：

- `display_name`
- `status`
- `category`
- `supports_time`

### 6.2 前端地图区域

`MapCanvas` 后续应围绕 `layer_id` 与 `render_type` 决定不同渲染方式：

- `raster` -> 栅格瓦片
- `vector` -> 矢量图层
- `point` -> 点图层
- `heatmap` -> 热力图层

### 6.3 前端时间轴

当前 `currentHour` 是 UI 原型字段，后续应转换为：

- `time_range.start_at`
- `time_range.end_at`
- `time_range.granularity`

### 6.4 结果面板

`InfoPanel` 后续建议直接绑定：

- 当前选中图层
- 当前任务状态
- 当前结果引用
- 诊断信息

## 9. 当前前后端现状梳理

### 9.1 前端现状

- 当前前端主路径已收口为 `2D-first demo`
- 地图、时间轴、图层侧栏、分析侧栏已形成稳定演示闭环
- 图层与时间联动目前主要由本地 `demo-adapter` 驱动
- 前端已经补了可对接后端的 `demo snapshot` 适配入口与统一 `runtime-api` 封装
- 当前前端尚未将主界面默认切换为真实后端数据拉取，目的是保证演示态稳定

### 9.2 后端现状

- 当前后端已提供：
  - `GET /layers`
  - `GET /demo/layers/snapshots`
  - `GET /demo/layers/{layer_id}/snapshot`
  - `POST /tasks`
  - `GET /tasks/{task_id}`
- 当前后端已新增交互总线骨架：
  - `POST /workflow-runs`
  - `GET /workflow-runs/{run_id}`
  - `GET /workflow-runs/{run_id}/events`
  - `PATCH /runtime/config`
  - `GET /runtime/status`

- 当前执行器、配置与事件流仍为内存骨架，后续替换为真实异步执行链路

## 10. 交互通道分层

### 10.1 核心工作流通道

用途：

- 前端向后端提交几乎所有核心处理指令
- 后端接收后形成统一 `run_id`
- 后续状态、日志、结果均围绕该 `run_id` 组织

接口：

- `POST /workflow-runs`
- `GET /workflow-runs/{run_id}`
- `GET /workflow-runs/{run_id}/events`

建议作为默认主通道，覆盖：

- 数据提交处理工作流
- 结果轮询
- 状态轮询
- 事件日志查询

### 10.2 配置更新通道

用途：

- 向后端提交基础设置、默认参数、运行时开关、provider 配置

接口：

- `PATCH /runtime/config`

特点：

- 频次不高
- 应与工作流参数分离
- 只提交“运行时配置”，不承担具体数据处理任务

### 10.3 日志与事件通道

用途：

- 向前端按需返回日志、状态变更、系统通知、数据准备完成提示

接口：

- `GET /workflow-runs/{run_id}/events`

事件通道：

- `status`
- `log`
- `data`
- `chart`
- `notification`
- `system`

说明：

- 当前第一版采用轮询
- 后续可平滑升级为 `SSE / WebSocket`

### 10.4 数据结果通道

用途：

- 向前端返回表格、JSON、地图图层引用、文件引用等数据结果

当前承载对象：

- `WorkflowResultReference`

当前返回位置：

- `GET /workflow-runs/{run_id}`

### 10.5 统计制图通道

用途：

- 向前端返回统计图、时间序列图、分布图、制图配置引用

当前承载方式：

- 作为 `WorkflowResultReference.result_kind = chart`
- 或作为 `events` 中的 `chart` 通道事件

### 10.6 后端状态反馈通道

用途：

- 提供系统总体状态、服务健康度、活跃任务数、当前配置快照

接口：

- `GET /runtime/status`

建议作为前端常用轮询接口之一

### 10.7 前端简单控制指令通道

用途：

- 发送懒加载、缓存清理、资源释放、目录重载、取消任务等轻量指令

接口：

- `POST /frontend/commands`

说明：

- 不负责承载重型数据处理
- 更适合界面控制与资源治理类请求

## 11. 建议补充与优化

你提出的 7 类通道已经基本完整，建议再明确补 3 层：

- `会话关联层`
  - 通过 `client_id / session_id / correlation_id / view_id` 把一次前端操作、一个工作流、若干日志和若干结果串起来
- `统一结果引用层`
  - 不要让图表、数据表、地图图层、文件下载分别长成不同返回格式
  - 建议统一收口到 `WorkflowResultReference`
- `统一事件流层`
  - 日志、状态、通知、图表准备完成、数据就绪都走 `WorkflowEvent`
  - 避免未来出现“每种功能一个单独回调接口”的碎片化问题

## 12. 推荐的使用方式

### 12.1 常用主路径

前端核心操作建议默认走：

1. `POST /workflow-runs`
2. `GET /workflow-runs/{run_id}`
3. `GET /workflow-runs/{run_id}/events`

### 12.2 低频配置路径

前端设置类操作建议走：

1. `PATCH /runtime/config`
2. `GET /runtime/status`

### 12.3 轻量控制路径

懒加载、清理、取消、刷新目录等建议走：

1. `POST /frontend/commands`

## 13. 当前阶段定位

当前这套接口还不是最终业务接口，而是：

- 一套前后端交互总线骨架
- 一套能承接真实算法执行器的统一入口
- 一套为后续 `services -> tasks -> algorithms/providers` 分层演进准备的稳定协议

下一阶段建议：

- 将 `workflow-runs` 接到真实 `services/tasks`
- 将 `runtime/config` 接到真实配置中心或数据库
- 将 `events` 从内存列表升级为可流式传输的事件机制
- 将 `result_refs` 接到真实算法输出、制图输出和文件输出
