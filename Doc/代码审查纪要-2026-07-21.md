范围：Open-Meteo 运行/数据拆分、Celery Beat 运维任务、前端加载与参数编辑、活文档对齐。

## 结论摘要

| 区域 | 判定 |
|------|------|
| 运行栈 `backend`（Redis/MinIO/`cgda-open-meteo`）vs 数据栈 `infra/data-sync` | **正确**；named volume 共享，不进项目目录 |

| 前端天气叠加 / WebGL / timeline | 主路径可用；coverage 轮询会闪 loading → **已 silent**；时间轴仅反映本地 coverage（钉 online 时可能不一致，已知限制） |

## 本轮已修复

1. **Celery 路由**：`task_routes` + Beat `options.queue` + 任务装饰器  
   - sync → `weather-batch`（soft 3600 / hard 3900）  
   - weather refresh → `weather-standard`  
   - timer tick → `standard`  
   - cleanup → `batch`  
2. **sync 前确保 named volume**（`external: true` 时不再静默失败）  
3. **HTTP trigger** `apply_async(queue=weather-batch)`  
4. **`getWeatherCoverage` `silent: true`**  
5. **ParamField number**：空值 / 非法值 → `null`，避免 `NaN`  
6. **文档**：`Doc/规范文档.md` §15.4、`本地Open-Meteo完善计划`、`data-sync/README`、`.env.open-meteo.example`

## 已知未改（可后续）

- 时间轴 coverage 始终探本地 OM；钉 `open-meteo-online` 时绿段可能与瓦片源不一致  
- coverage 探针 `Asia/Shanghai` vs 浏览器本地时区可能偏差  
- sync 无全局互斥锁（Beat + UI + `launch.py sync` 可并行写 volume）  
- `_LOCAL_SYNC_JOBS` 仅内存，FastAPI 多进程不共享  

## 建议手工回归

```text
python launch.py restart   # 或 stop + start，使 Beat/Worker 加载新路由
设置页 → Open-Meteo → 立即同步 → 状态应变 SUCCESS（非长期 PENDING）
时间轴绿段 / 风场粒子 / 温度标量层
工作流 Inspector 数值参数清空不应写入 NaN
```
