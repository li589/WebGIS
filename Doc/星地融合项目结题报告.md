## 1. 项目背景与目标

### 1.1 背景

地理信息与遥感应用场景中，课题组长期面临三类割裂：

1. **数据割裂**：卫星遥感（SMAP、FY-3、MODIS/VIIRS、GEE 等）与地面站点（ISMN、CASMOS、生态站网等）分散在不同介质与格式中；
2. **计算割裂**：MATLAB/Python 科研脚本与 Web 展示系统各自为政，缺少统一任务入口与结果回传；
3. **展示割裂**：分析结果多以本地文件或截图交付，难以在统一地图舞台上完成时空浏览、叠加对比与交互出图。

本项目面向上述痛点，建设一套**星地融合**的综合地理数据平台：将多源数据接入、工作流编排、异步科学计算、结果产物管理与 WebGIS 可视化闭环串联，形成「取数 → 计算 → 出图」一体化能力。

### 1.2 建设目标

| 目标维度 | 具体要求 |
| -------- | -------- |
| 一体化 | 同一 Web 入口完成数据流水线工作与地图出图展示 |
| 星地融合 | 卫星遥感 + 地面站点 + 公共气象/开放数据统一接入与协作计算 |
| 工作流驱动 | 可视化节点编排（类 ComfyUI），后端 DAG 执行与状态回传 |
| 协议先行 | 前后端与算法包围绕统一契约协作，降低字段漂移 |
| 可扩展 | 多课题组算法以 Provider/模块方式插件化接入 |
| 工程化 | 异步队列、对象存储、配置真源、瓦片数据面与可观测性具备可运行实现 |

### 1.3 项目定位（一句话）

> 面向科研与业务演示的**星地融合 WebGIS 平台**：以工作流编排驱动数据流水线，以双通道 API 与统一瓦片契约驱动地图出图展示。

## 2. 系统总体架构

### 2.1 三层架构

系统采用清晰的三层职责划分：

```text
┌─────────────────────────────────────────────────────────────┐
│  展示层（Frontend）                                           │
│  Vue 3 + TypeScript + MapLibre GL + Canvas/WebGL 叠加         │
│  地图主舞台 · 图层/时间轴 · LiteGraph 工作流 · 设置与出图     │
└───────────────────────────┬─────────────────────────────────┘
                            │ REST（控制流） + 瓦片/Artifact（数据流）
┌───────────────────────────▼─────────────────────────────────┐
│  服务层（Backend）                                            │
│  FastAPI + Celery/Redis + SQLite + MinIO                      │
│  workflow-runs · weatherengine · unified-tiles · GEE · 配置面 │
└───────────────────────────┬─────────────────────────────────┘
                            │ Provider Bridge / Job 契约
┌───────────────────────────▼─────────────────────────────────┐
│  计算层（Algorithms）                                         │
│  Python 工程化算法包                                          │
│  data_access · modules · workflow · publish · storage         │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 仓库与工程分层

```text
Comprehensive Geographic Data Analysis system/
├─ Doc/          # 方案、技术栈、规范、决策与结题文档
├─ Env/          # 本地开发环境辅助
├─ Code/         # 工程代码
│  ├─ frontend/  # WebGIS 展示与交互
│  ├─ backend/   # API、编排、天气引擎、瓦片、GEE
│  ├─ algorithms/# Python 算法包与数据接入
│  ├─ shared/    # 共享协议契约
│  ├─ infra/     # 数据面 compose（如 open-meteo-sync）
│  └─ docs/      # 实现向协作文档
├─ Tools/        # 数据下载与校验工具
├─ Example/      # 外部参考（如 Windy 风格）
└─ launch.py     # 一键启停（Docker + API + Workers + 前端）
```

### 2.3 双通道接口设计（核心架构原则）

为避免「状态轮询」与「大体量结果」互相拖累，后端 API 明确划分为：

| 通道 | 职责 | 典型接口 |
| ---- | ---- | -------- |
| **控制流** | 任务提交、状态、事件、取消/重试、runtime | `POST/GET /workflow-runs`、`/events`、`/cancel`、`/retry`、`/runtime/*` |
| **数据流** | 结果视图、artifact、预览图、瓦片按需加载 | `/workflow-runs/{id}/view`、`/artifacts/*`、`/weather/tiles/*`、`/unified-tiles/*` |

设计收益：

- 控制流轻量、适合高频轮询与状态机；
- 数据流以展示为中心，按需加载栅格/矢量资源；
- 前端消费稳定 view model（如 `WorkflowRunViewResponse`），不直接耦合后端内部 DTO。

### 2.4 基础设施双栈

| 栈 | 组成 | 用途 |
| -- | ---- | ---- |
| 运行栈 | Redis、MinIO、Open-Meteo API（`backend/docker-compose.yml`） | 队列、对象存储、在线/本地气象服务 |
| 数据栈 | `infra/data-sync`（如 open-meteo-sync） | 一次性/批量数据同步，与运行栈隔离 |

一键联调：`python launch.py start | sync | stop | status | flush`。

## 3. 前端展示与出图能力

### 3.1 技术选型与定位

- **框架**：Vue 3 + TypeScript + Vite + Pinia
- **地图主路径**：MapLibre GL JS（**2D-first**）
- **叠加渲染**：Canvas / WebGL（风粒子、标量场、等值线等）
- **工作流编辑**：LiteGraph 可视化画布
- **3D**：Cesium / vue-cesium 依赖已引入，尚未成为默认主链

前端定位为「展示层 + 交互层」，而非单纯地图页：统一承载底图、天气叠加、图层树、时间轴、工作流状态、信息面板与截图导出。

### 3.2 出图与交互能力清单

| 能力 | 说明 |
| ---- | ---- |
| 底图与行政区 | MapLibre 底图切换；行政区边界叠加 |
| 天气标量场 | 全球对齐格网 + 标准 z/x/y 天气瓦片；LUT 着色、时次交叉淡入 |
| 风场可视化 | 粒子流（WebGL/Canvas）、风羽、流线/等值线 |
| 图层管理 | 分类、搜索、拖拽排序、批量显隐；天气源钉选（多 Provider） |
| 时间轴 | 时空结果 scrubbing / 播放控制 |
| 工作流编辑器 | 画布编排 → 编译 → 提交执行；状态面板与结果信息面板 |
| 导入导出 | 栅格/矢量导入；截图与 PDF 导出 |
| 系统设置 | API Key、数据源、远程存储、天气 Provider、缓存清理等 |

### 3.3 天气渲染工程要点（展示侧主成果）

天气展示已从「整图拉取」演进为**视口驱动的标准瓦片热路径**：

1. `weather-tile-manager` 按视口计算瓦片集合、并发与预取；
2. `GET /weather/tiles/{layer_id}/{z}/{x}/{y}` 拉取 GeoJSON 格网（可 `provider` 钉源）；
3. 格心归属与半开裁点前后端一致，消除格元空隙与叠框；
4. 父级缩放级仅作空洞垫底；合并缓存带覆盖签名；
5. 风粒子与标量场支持密度、拖尾、不透明度等商业化视觉参数。

底图栅格走独立入口 `GET /unified-tiles/...`，与天气瓦片严格分流，避免契约混用。

### 3.4 前端模块结构（摘要）

```text
frontend/src/
├─ views/DashboardView.vue          # 主布局编排
├─ components/
│  ├─ map/                          # 地图、天气 overlay、风场、标量 WebGL
│  ├─ workflow/                     # LiteGraph 编辑器、状态、定时器
│  ├─ settings/                     # 系统设置各 Tab
│  ├─ LayerSidebar / Timeline* / InfoPanel / ModeToolbar
├─ stores/                          # UI、图层、天气瓦片、工作流、设置
└─ services/                        # runtime / weather-tile / settings / workflow API
```

## 4. 后端编排与服务能力

### 4.1 定位

后端核心不是 CRUD，而是稳定的**任务执行链**：

```text
API routers → workflow services → Celery tasks / bridges → providers
         → artifact / tiles / view model → 前端
```

原单体 `interaction_hub` / 巨型 `routes` 已拆分为按域路由与聚焦服务，便于长期维护。

### 4.2 主要能力域

| 域 | 能力 |
| -- | ---- |
| 工作流运行 | 提交、查询、事件流、取消、重试；`retry_pending` 与 `FailureCategory` |
| 工作流定义 | CRUD、节点模板、LiteGraph → `workflow_definition` 编译 |
| 天气引擎 | 点查、网格预报、标准天气瓦片；多 Provider（open-meteo-online/local、WeatherAPI、OpenWeather） |
| 统一瓦片 | 底图代理与缓存统计；天气与底图入口分离 |
| 配置面 | `/config/*`：API Key、数据源、远程存储、GEE、天气 Provider、静态缓存 |
| 算法桥接 | Python Provider bridge；artifact 与 preview.png |
| GEE | Earth Engine 服务端代理模块可挂载 |
| 容量治理 | business 池与 weather_tile 池双上限，避免瓦片热路径挤占分析任务 |

### 4.3 执行与存储事实

| 项 | 现状 |
| -- | ---- |
| 同步/异步 | `sync` / `celery` 双模式 |
| 状态持久化 | SQLite（PostGIS 为后续目标） |
| 对象存储 | MinIO + 本地双后端抽象 |
| 缓存/队列 | Redis（任务队列、天气缓存、限流与断路器） |
| 鉴权 | `/config` 写操作与栅格导入需 `X-API-Key`；密钥 env + DB 覆盖真源 |

## 5. 算法包与星地数据流水线

### 5.1 算法包工程化结构

`Code/algorithms/providers/Python/` 已由脚本集合演化为可被平台调度的工程包：

| 目录 | 职责 |
| ---- | ---- |
| `contracts` / `interfaces` | 任务、数据、产品、事件与适配接口 |
| `runner` / `workflow` / `service` | 统一入口、DAG 执行、HTTP/队列接入 |
| `data_access` | 发现、解析、物化（local/HTTP/SMB/SFTP/MinIO/Cache） |
| `modules` | 原生算法与数据节点（SMAP/NDVI/FY/station/omega 等） |
| `ingest` / `algorithms` / `publish` | 读入装配、数值核心、产品与 manifest |
| `pipelines` | 历史兼容层（主方向已收敛到 modules + workflow） |

### 5.2 星地融合数据谱系

**卫星与再分析侧（示例）**

| 数据 | 用途 |
| ---- | ---- |
| SMAP L3 土壤水分 | 被动微波土壤水分 |
| FY-3 MWRI 亮温 | 国产微波成像 |
| VIIRS/MODIS NDVI、土地覆盖 | 植被与地表分类 |
| ECMWF 风场、融合降水 | 气象辅助 |
| GOSAT XCO2 等 | 大气产品 |
| GEE 影像集合 | 云端渲染/按需计算 |

**地面侧（示例）**

| 数据 | 用途 |
| ---- | ---- |
| ISMN / CASMOS / 生态站网 | 土壤水分与温度验证、同化与对比 |

**格式枢纽**：`.mat` / HDF / NetCDF 等科研格式经算法包转换为前端可消费的 COG、PNG 预览、GeoJSON、manifest 等。

### 5.3 可视化数据流水线节点（已落地）

画布侧支持「下载 → 解压 → 配置读取 → 变量提取 → 格式转换 → 出图层」链路，典型节点包括：

| 节点 | 作用 |
| ---- | ---- |
| `remote_fetch` / `http_open_data` | 远程/开放数据物化下载 |
| `archive_extract` | zip/tar 等解压 |
| `config_read` | JSON/YAML/INI/XML 配置解析 |
| `variable_extract` | 通用变量读取 |
| `format_convert` | 格式转换 |
| `output_map_layer` | 登记为地图图层产物 |

前端 Run 路径：LiteGraph 序列化 → `POST /workflow-definitions/compile` → `workflow-runs` 执行 → 结果挂载地图。

### 5.4 典型端到端链路

```text
多源数据（星/地/开放数据/NAS）
        ↓ data_access materialize
算法模块 / 课题组工作流（omega、NDVI、station、…）
        ↓ publish + artifact
后端 view / layer_assets / tiles
        ↓
MapLibre + Overlay 出图展示
```

## 6. 关键设计原则与工程治理

### 6.1 设计原则

1. **前后端分离、协议先行**：共享契约 + OpenAPI 类型生成；
2. **控制流 / 数据流分离**：状态轻、数据重，各自优化；
3. **算法插件化**：课题组不各自定义请求/结果格式；
4. **数据与项目解耦**：算法只消费 PreparedInput，不绑死存储介质；
5. **先主链后扩展**：优先稳定 workflow / 天气瓦片 / provider 桥接，再补 PostGIS、专用切片服务、Cesium 3D、Nginx。

### 6.2 文档治理

文档按六层管理：入口 → 架构 → 协议 → 执行 → 专题 → 附录，保证「当前事实」与「历史试验」不互相污染。主入口以各 README 与决策纪要为准；带日期文档作阶段快照。

### 6.3 工程收口成果（摘要）

截至 2026-07-21，已完成或基本完成：

- 配置真源（env + DB 覆盖）与写操作鉴权；
- Legacy/demo 路由默认关闭；OpenAPI drift 检查；
- 天气瓦片与底图瓦片契约锁定；
- HTTP/MinIO 真实 materialize（不再假 ready）；
- 远程存储（SFTP/SMB 等）设置面与凭证库；
- 天气渲染几何一致性与 WebGL 视觉增强；
- 课题组画布编译执行 + 数据获取节点全链路。

## 8. 项目成果总结

### 8.1 已交付核心成果

1. **一体化 Web 平台**：同一系统完成数据流水线编排与时空出图；
2. **星地数据接入框架**：卫星/站点/开放数据/远程 NAS 统一 data_access 与画布节点；
3. **工作流主链**：`workflow-runs` + LiteGraph 编译执行 + 事件/取消/重试；
4. **天气出图主链**：多 Provider 天气引擎 + 标准瓦片 + Windy 风格风场/标量可视化；
5. **双通道 API 与共享契约**：控制流与数据流分离，前后端契约可校验；
6. **基础设施可运行栈**：Redis / MinIO / Open-Meteo + `launch.py` 一键联调；
7. **工程治理体系**：文档分层、配置真源、安全收口、测试与 OpenAPI 漂移检查。

### 8.2 对课题组业务的价值

- **缩短路径**：从「本地脚本算完再手工制图」变为「画布编排 → 异步计算 → 地图直接出图」；
- **统一入口**：多数据源、多算法模块在同一平台注册与调度；
- **可演示可扩展**：既支持导师演示与联调，也为后续超算/MinIO 部署预留存储抽象；
- **星地对照**：卫星产品与地面站点可在同一时间轴与图层体系下叠加分析。

### 8.3 已知边界与后续工作

| 类别 | 内容 |
| ---- | ---- |
| 主链未启用 | Cesium 3D 默认模式、deck.gl 大规模点云叠加 |
| 基础设施 | PostGIS、Nginx 全站、Martin/TiTiler 专用切片 |
| 数据环境 | 真实课题组端到端绿测、实机 NAS 联调仍依赖现场数据环境 |
| 前端债 | Layers 大 store 继续拆分等工程优化 |
| 产线 | GEE 可观测性与部分算法产线仍在完善 |

上述项不削弱本期「流水线 + 出图一体化」主目标的达成，而属于明确规划的下一阶段平台增强。

## 9. 结题结论

本项目已建成一套面向**星地融合**场景的综合地理数据分析 Web 系统：以 **Vue/MapLibre 展示层**、**FastAPI/Celery 编排层**、**Python 算法包计算层**构成稳定三层架构；以 **双通道 API** 与 **工作流 DAG** 贯通「数据获取—科学计算—地图出图」全链路；并在天气瓦片渲染、课题组数据节点、远程存储与配置治理等方面完成工程化落地。

相对立项时的方案与原型目标，系统已具备：

- 可运行的一体化联调与演示能力；
- 可扩展的多源数据与多算法接入框架；
- 可维护的契约、文档与安全收口基础。

因此，就「用于数据流水线工作与出图展示的一体化 Web 项目」这一结题定位而言，**本期建设目标已达成**；PostGIS、3D 主链、专用瓦片服务与真实数据环境全量绿测等，建议作为下一阶段持续建设内容。
